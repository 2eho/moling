# 墨灵 · 章节灵感卡组合生文算法 v2.0

> 基于 C3TG、StoryWriter、LangChain、NovelForge、Multi-Prompt Learning 五大框架 + 混合流水线架构（传统算法 + LLM）的综合算法设计

## 一、问题定义

### 1.1 核心场景

```
抽卡 → [3张章节灵感卡] → 用户选择 → 组合生文

可选模式:
  不选 / 单卡 / 双卡 / 全选 / 选卡+手动输入
```

### 1.2 卡片数据结构

```
章节灵感卡 {
  id: string
  rarity: "common" | "rare" | "epic" | "legendary"
  rarity_weight: 1 | 2 | 3 | 4

  // 叙事方向
  direction_text: string
  direction_type: "稳妥" | "有趣" | "惊艳" | "神之一手"

  // 4 库引用（生成时由动态层+4库注入上下文后生成）
  characters: [{ id, name, state_requirement }]
  plot_promises: [{ id, title, advance_type }]
  timeline_point: string
  world_rules: [{ id, rule, constraint }]

  // 动态层关联
  current_story_state: string
  unresolved_hooks: string[]
  dynamic_conflict_score: number
}
```

### 1.3 系统数据组件

```
上下文体系
│
├── 📋 动态层（每章更新 · 流动事实 · 故事坐标系）
│   ├── 前情摘要       → "故事到哪了"的自然语言总结
│   ├── 章节锚点       → POV / 地点 / 时间
│   ├── 连贯性基线     → must_hold / must_not（硬约束）
│   ├── 未收束钩子     → 未解决的悬念 Top 10
│   ├── 最近3章变更    → 增量日志
│   └── 🔐 秘密矩阵    → 信息不对称（谁知道了什么 / 谁还不知道）
│
├── 📚 四库（跨章不变 · 静态真理 · 世界设定）
│   ├── 👥 人物库      → 人物设定、关系、状态机
│   ├── ⏱️ 时间线库    → 绝对时间线与事件序列
│   ├── 📖 剧情承诺库  → 伏笔、弧光、子情节
│   │                   · event_type 标注 → 🏥健康监控
│   └── 🌍 世界观库    → 世界规则、阵营、势力
│
└── 🃏 卡片方向（用户选择 · 创意增量 · 每章不同）
```

#### 1.3.1 秘密矩阵 vs 未收束钩子的关系

> **关键区分：** 两个概念容易混淆，但在系统中有明确分工。


| 维度         | 🔐 秘密矩阵（`information_asymmetry`） | 🪝 未收束钩子（`unresolved_hooks`） |
| ------------ | -------------------------------------- | ----------------------------------- |
| **视角**     | **角色之间**的信息差                   | **读者**的悬念期待                  |
| **主体**     | 谁知道了什么 × 谁还不知道             | 什么情节还没有结果                  |
| **生命周期** | 随章节更新`known_by/unknown_to`        | 随"被解决"而移除                    |
| **存储位置** | 动态层`information_asymmetry`          | 动态层`unresolved_hooks`            |
| **冲突检测** | 角色说了不该知道的话 → 🚫 阻止        | 章节没解决该解决的悬念 → ⚠️ 提醒 |
| **示例**     | "林夜不知道叶灵是卧底"                 | "玉佩的真实来历还没有揭示"          |

> **协同工作示例：** 一个"秘密"被揭露后，对应的"钩子"也会被标记为已解决。秘密是**角色的认知边界**，钩子是**读者的阅读期待**。

### 2.3 为什么用混合架构？


| 维度               | 纯 LLM 方案 | 混合方案（传统+LLM）      |
| ------------------ | ----------- | ------------------------- |
| LLM 调用次数       | 7-8 次/章   | **4 次**/章（含异步收纳） |
| 大模型调用         | 4-5 次      | **1 次**（仅正文写作）    |
| 单章延迟（同步）   | ~35-50s     | **~12-20s**               |
| 单章延迟（含异步） | —          | **+2-3s**（不阻塞用户流） |
| 单章成本           | 高          | **降低 60%+**             |
| 前 6 步可缓存      | ❌ 每次重跑 | ✅ 调权重不重跑           |
| 可调试性           | 黑盒        | ✅ 前 6 步单步调试        |
| 结果确定性         | 概率性      | ✅ 前 6 步确定            |

### 2.4 置信度评分与 Fallback

传统算法不是永远正确的。当它的判断置信度低时，fallback 到 LLM：

```
传统算法输出一个评分:
  冲突评分 ∈ [0, 1]

  > 0.7 → 高置信度 → 直接采用
  [0.3, 0.7] → 中等置信度 → 标记为"建议"，但采用
  < 0.3 → 低置信度 → fallback 到 LLM

编织评分 ∈ [0, 1]:
  > 0.8 → 直接采用规则引擎的结果
  ≤ 0.8 → fallback 到 LLM 选编织模式
```

这保证了 80% 的常规情况走快路径，20% 的复杂/模糊情况走 LLM 兜底。

### 2.8 🔐 秘密生命周期（信息不对称矩阵）

> **核心问题**：角色知道了他不该知道的事 / 角色不知道他应该知道的事 → 连续性崩溃。
> **解决方案**：在动态层中追踪每个"秘密"的 `known_by` 和 `unknown_to`，每章自动检测不对称状态。

```
                     秘密的完整生命周期
                   
         ┌─────────────────────────────────────────────────────┐
         │              ① 秘密埋设（Phase 4 提取）              │
         │  输入: 正文中出现的"角色A知道某事，角色B不知道"        │
         │  输出: SecretEntry {                                 │
         │    id, secret_text, created_chapter,                │
         │    known_by: [A], unknown_to: [B,C,D],              │
         │    secrecy_level: "absolute" | "partial" | "open"   │
         │  }                                                  │
         └──────────────────────┬──────────────────────────────┘
                                │
                                ▼
         ┌─────────────────────────────────────────────────────┐
         │              ② 秘密传播（Phase 4 每章增量）          │
         │  LLM 从新章节中检测:                                │
         │  • 角色B发现了秘密 → known_by += [B]                 │
         │  • 角色C听到对话 → known_by += [C]                   │
         │  • 秘密被公开提及 → secrecy_level = "open"           │
         │  • 秘密被证实为假 → marked as "debunked"             │
         └──────────────────────┬──────────────────────────────┘
                                │
                                ▼
         ┌─────────────────────────────────────────────────────┐
         │              ③ 冲突检测（§3.3 动态层冲突检测）        │
         │  当卡片方向 / 用户输入 涉及秘密时:                   │
         │  • 角色A打算告诉B一个B还不知道的秘密 → ✅ 允许      │
         │  • 角色B在对话中说出了TA还不知道的秘密 → ❌ 冲突阻止 │
         │  • 角色A欺骗了B（A知道但B不知道） → ℹ️ 标记为合理   │
         │                                                     │
         │  冲突判定逻辑:                                       │
         │  生成正文中 → 正则/实体匹配检测 →                    │
         │    若 角色P 提及/使用/反应了 秘密S 的内容 →          │
         │      若 P ∈ S.unknown_to → 🚫 高亮冲突段落          │
         │      若 P ∈ S.known_by → ✅ 继续                     │
         └──────────────────────┬──────────────────────────────┘
                                │
                                ▼
         ┌─────────────────────────────────────────────────────┐
         │              ④ 秘密揭露 / 收束                        │
         │  • 全部角色都知道 → secrecy_level = "open"           │
         │  • 从动态层归档到 vault_changelog (永久记录)          │
         │  • 不再参与冲突检测（减少不必要的计算）                │
         │  • 可触发剧情承诺库的"advance"事件（如果该秘密关联    │
         │    某个活跃承诺）                                     │
         └─────────────────────────────────────────────────────┘
```

#### 2.8.1 四层知识层级（信息不对称理论）

> 来源：webnovel-writer 信息不对称理论 + 叙事学 "知识层级" 模型

```
层级          知道什么                          叙事价值
─────────────────────────────────────────────────────────
L0: 读者    知道所有公开信息 + 部分秘密     阅读体验的基础
L1: 主角    知道最多（通常比读者少一点）    代入感来源
L2: 配角    知道部分（通常围绕自身主线）    推进情节的工具
L3: 反派    知道关键秘密的不同侧面           悬念制造的来源

每层之间的"信息差"就是秘密的"债务"——
债务越大，揭露时的情感冲击越强。
```

#### 2.8.2 秘密债务模型

```python
# 每次秘密存续（未被揭露），"债务"持续增长
debt = (current_chapter - secret.created_chapter) × len(secret.unknown_to)

# 债务 > 阈值时，触发"建议揭露"通知
if debt > 30:  # 假设 10 章 × 3 个角色不知 = 30
    → UI 提示"秘密【xxx】的信息债务过高，建议安排揭露"
```

> **用债务代表什么？** 一个秘密拖得越久、不知道的人越多，读者就越着急。债务模型量化了这种"读者想知道答案"的迫切程度。当债务过高时，不是"出错了"，而是"时机成熟了"——应该安排揭秘场景。

### 3.2 为什么动态层必须进？

**它是"故事坐标系"，不是"信息补充"。**

```

❌ 无动态层:
Prompt = "写第15章，方向是林夜发现装置异常"
AI输出 = "林夜走进研究所……"
→ 研究所第14章结尾已被炸毁！AI 不知道。

✅ 有动态层:
Prompt = "【当前状态】研究所地下三层已在第14章被炸毁。
林夜正在废墟中搜索。时间: 深夜23:47。"
AI输出 = "林夜拨开碎石，在废墟深处发现了装置的微光……"
→ 准确衔接。

```

**学术依据：**
- Context Amnesia（arXiv 2310.00598）：AI 不会自动记住前文
- Lost in the Middle（Liu et al. 2023）：prompt 中部信息召回率显著下降，所以动态层必须放在 prompt 头部

### 3.3 为什么四库必须进、但绝不能全量进？

**论点 A：只靠卡片元数据不够**

卡片只有引用 `plot_promises: [{id: "identity_mystery", title: "林夜身世之谜"}]`。AI 知道**有一个承诺要推进**，但不知道它**具体是什么**。

```

无 4 库注入 → AI 每次凭记忆编造承诺内容
→ 各章之间细节逐渐矛盾
→ 第30章时已面目全非

```

**论点 B：全量注入有害**

30 章后 4 库全量 ≈ 5000+ 字。在 8000 token 窗口里挤满后：
- 中间段信息被 AI 遗忘的概率剧增
- 你费劲塞进去的规则，AI 根本"没看见"
- 生成质量反而下降

**论点 C：零注入 = 慢性死亡**

每个错误单独看不大，但累积到第 30 章时故事已偏离十万八千里。

**结论：必须卡片引用过滤 + 层级压缩。**

### 3.4 四库过滤算法

```

输入: 用户选择的卡片集合 S
输入: 4 库完整数据

Step 1: 提取卡片引用的所有 ID
character_ids   = ∪_{c∈S} c.characters.map(c => c.id)
promise_ids     = ∪_{c∈S} c.plot_promises.map(p => p.id)
timeline_points = ∪_{c∈S} c.timeline_point
world_rule_ids  = ∪_{c∈S} c.world_rules.map(r => r.id)

Step 2: 按 ID 提取全文
人物库:  只提取 character_ids 的人物（≤300字/人）
时间线库: 只提取 timeline_points ±3 条事件
剧情承诺库: 只提取 promise_ids 的承诺全文
世界观库: 只提取 world_rule_ids 的规则原文

Step 3: 层级压缩
Level 1（默认）: 过滤数据全文注入
Level 2（30章后）: 人物只保留"当前状态"字段

```

### 3.6 编织模式模板定义

编织模式决定了 Layer 3 注入的**叙事结构指令**。LLM 不止收到一个标签名，而是收到结构化节拍指引。

#### 3.6.1 因果链（Cause → Effect → Revelation）

适用于：两卡有因果关系（"A 导致 B"），或全选三卡构成因果链

```

编织模式: 因果链

【注入 Layer 3 的 Prompt 指令】
本章按以下三段式结构推进：
第一段【因】: 角色行动或事件触发（展示原因）
第二段【果】: 触发后的连锁反应（推进后果）
第三段【揭示】: 新的信息/认知浮现（为下一章埋钩子）

结构要求:

- 开篇直接从前章结尾接续（不重复说明），节奏紧凑
- 三段之间用"因为→所以→由此得知"的逻辑链连接
- 每段结尾自然引出下一段（不用分隔符）
- 结尾钩子必须是对下一章产生直接影响的揭示

【大纲填充模板】
part1: 因 · 35%篇幅 | 角色行动/事件触发
part2: 果 · 40%篇幅 | 连锁反应/后果推进
part3: 揭示 · 25%篇幅 | 新信息浮现/章尾钩子

```

#### 3.6.2 平行交替（Dual Narrative Interweaving）

适用于：两卡各自独立但主题相关（如双 POV、AB 线并行）

```

编织模式: 平行交替

【注入 Layer 3 的 Prompt 指令】
本章有两条平行叙事线交替推进：
A线（主）: {主卡方向文本}
B线（副）: {副卡方向文本}

结构要求:

- 以 A1 → B1 → A2 → B2 → A3 → 交汇 的节奏推进
- 每条片段 300-500 字，交替 ≥ 2 轮
- A线和B线在章末交汇（角色/事件/主题上的关联）
- 交汇点必须是两条线共同推进同一个剧情承诺
- 交替时用时间/地点标注切换，避免读者混淆

【大纲填充模板】
segment 1: A线第一段 · 20%篇幅
segment 2: B线第一段 · 20%篇幅
segment 3: A线第二段（推进）· 20%篇幅
segment 4: B线第二段（推进）· 20%篇幅
segment 5: 双线交汇 · 20%篇幅 → 章尾钩子

```

#### 3.6.3 主线+支线（Main + Side Quest）

适用于：一卡为主方向、另一卡为辅（权重差距 ≥ 20%）

```

编织模式: 主线+支线

【注入 Layer 3 的 Prompt 指令】
本章包含一条主线推进 + 一条支线点缀：
主线（权重 {mainWeight}）: {主卡方向文本}
支线（权重 {sideWeight}）: {副卡方向文本}

结构要求:

- 主线占 70% 篇幅，推进核心剧情承诺
- 支线占 30% 篇幅，在以下位置之一插入：
  a. 主线中场休息时（情绪低谷段）
  b. 主线动作场景后（节奏缓冲）
  c. 章末支线突然与主线交叉（转折点）
- 支线必须服务于以下至少一个目的：
  · 展示世界观侧面
  · 铺垫未来关键信息
  · 深化配角形象
- 支线不可喧宾夺主（不超过连续 400 字）

【大纲填充模板】
part1: 主线启动 · 25%篇幅
part2: 支线插入 · 15%篇幅
part3: 主线推进 · 30%篇幅
part4: 主线高潮/冲突 · 20%篇幅
part5: 支线回扣/章尾钩子 · 10%篇幅

```

#### 3.6.4 编织模式选择规则

```

输入: 用户选择的 2-3 张卡 + 权重分配 + 相容性评分

[ T ] 选择逻辑:

双卡模式（Mode 2）:
- 两卡权重差距 ≥ 20% → 主线+支线
- 两卡权重接近（差距 < 20%）且同一 POV → 因果链
- 两卡权重接近且不同 POV/场景 → 平行交替
- 置信度 < 0.3 → [F] LLM 选模式（提示：输出 causal | parallel | main_side）

全选三卡（Mode 3）:
- 三张卡有明确时序关系 → 因果链扩展（因→果→转折）
- 角色空间分布不同 → 平行交替+交汇（三线汇一）
- 一主二副 → 主线+支线（双支线合并为一条 30% 支线）

单卡（Mode 1）/ 手动输入（Mode 4）:
- 无编织模式（仅一段式）

```

## 五、连贯性保障机制

### 5.1 生成前校验（Pre-check）

```
[T] 检测项目:
┊ 人物状态一致性（vs 人物库 filtered）
┊ 时间线连续性（vs 时间线库 filtered）
┊ 剧情承诺推进逻辑（vs 承诺库 filtered）
┊ 世界观规则遵守（vs 世界观库 filtered）
┊ 与前情摘要的衔接（vs 动态层）
┊ 连贯性基线一致（vs 动态层基线）

通过率 < 70% → 提示用户可能的风险
通过率 < 40% → 建议降低卡数量或换卡

```

### 5.2 生成后校验（Post-check）

```
[T] ① 事实一致性检查
人物状态 vs 人物库 → 偏差标记
[T] ② 剧情承诺推进检查
是否推进了所选卡标记的承诺 → 未推进标记
[T] ③ 时间线一致性
事件顺序 vs 时间线库 → 矛盾标记
[T] ④ 世界观规则检查
内容 vs 规则原文 → 违反自动修正
[T] ⑤ 动态层基线校验
是否违反 must_hold/must_not → 定位段落自动重写
[T] ⑥ 未收束钩子状态更新
已解决→移除 / 新悬念→添加
[T] ⑦ 🔐 秘密一致性与债务检查
角色是否说出了TA不知道的秘密 → 冲突标记
角色是否对已知秘密做出了矛盾反应 → 冲突标记
秘密债务 > 30 → 建议安排揭露（非阻塞提示）

```

### 5.3 子情节健康监控（Anti-Drift · 持续运行）

> **核心问题**：子情节（伏笔/弧光）被"遗忘"——作者写到后面忘了前面埋的线。
> **解决方案**：3 条 SQL 规则 + 1 条元规则，每次确认收纳/开启下一章时自动检测。
> **特别设计**：所有规则**不依赖 LLM**，纯 SQL/算法，零额外成本。

#### 5.3.1 数据基础：Advancement 事件类型扩展

现有 `advancement_log` 从 `{chapter, note}` 扩展为 `{chapter, note, event_type}`：

```python
# Noveble 5 事件类型
event_type: Literal[
    "foreshadow",  # 伏笔：埋设新线索/暗示
    "advance",     # 推进：逐步揭示/发展
    "twist",       # 转折：意外变化/反转
    "climax",      # 高潮：关键冲突/爆发
    "wrap"         # 收束：回收/完结
]
```
> 为什么是这 5 类？来自 Noveble 叙事结构理论——任何子情节的推进都可以归入这 5 种事件类型之一。这保证了**分类覆盖率 100%**，不会有 LLM 无法判断的边界情况。

#### 5.3.2 三条健康规则

```sql
-- R1: 8 章未推进告警（🟡 黄色）
-- 触发条件：一个活跃/推进中的承诺，最近 8 章内没有任何推进记录
-- 处理方式：UI 黄色横幅提示"伏笔【xxx】已 8 章未推进"
SELECT pp.title, pp.planted_chapter, MAX((log->>'chapter')::int) AS last_advance
FROM vault_plot_promises pp
CROSS JOIN LATERAL jsonb_array_elements(pp.advancement_log) AS log
WHERE pp.project_id = $1
  AND pp.status IN ('active', 'advancing')
  AND pp.planted_chapter < $current_chapter - 8
GROUP BY pp.id
HAVING MAX((log->>'chapter')::int) < $current_chapter - 8
   OR MAX((log->>'chapter')::int) IS NULL;

-- R2: 4+ 同类型重复推进行为（🟠 橙色）
-- 触发条件：一个承诺 >= 4 条推进记录，但所有记录都是同一 event_type
-- 含义：子情节在"原地踏步"——比如连续 4 次都是"伏笔"，没有推进
-- 处理方式：UI 橙色警告"【xxx】连续 4 次同类推进，建议加入转折或收束"
WITH type_counts AS (
    SELECT pp.id, pp.title, log->>'event_type' AS et, COUNT(*) AS cnt
    FROM vault_plot_promises pp
    CROSS JOIN LATERAL jsonb_array_elements(pp.advancement_log) AS log
    WHERE pp.project_id = $1 AND pp.status IN ('active', 'advancing')
    GROUP BY pp.id, log->>'event_type'
), total AS (
    SELECT id, COUNT(*) AS type_count, SUM(cnt) AS total_count
    FROM type_counts GROUP BY id
)
SELECT t.id, t.title, t.total_count
FROM total t
WHERE t.total_count >= 4 AND t.type_count = 1;

-- R3: 10 章静默承诺告警（🔴 红色）
-- 触发条件：超过 10 章既无推进、在最新章节中也无提及
-- 含义：子情节被"完全遗忘"
-- 处理方式：UI 红色横幅 + 建议"考虑回收或标记为废弃"
-- 注：R3 = R1 条件 + 额外检查近 3 章草稿文本中是否提及该承诺的关键词
-- 如果有关键词提及但无正式推进记录 → 降级为 R1 级别
SELECT pp.title, pp.planted_chapter
FROM vault_plot_promises pp
WHERE pp.project_id = $1
  AND pp.status IN ('active', 'advancing')
  AND pp.planted_chapter < $current_chapter - 10
  AND NOT EXISTS (
      SELECT 1 FROM jsonb_array_elements(pp.advancement_log) AS log
      WHERE (log->>'chapter')::int > $current_chapter - 10
  )
  -- 降级检查：近 3 章草稿有关键词提及（由应用层从 draft_chapters 查询）
  AND NOT EXISTS (
      SELECT 1 FROM chapters c
      WHERE c.project_id = $1
        AND c.status = 'draft'
        AND c.chapter_number > $current_chapter - 3
        AND c.content ILIKE '%' || pp.title || '%'
  );
```

#### 5.3.5 与现有系统的集成

```
生成流水线中的触发点:
  Phase 3 完成后（动态层更新后）→ 健康检查 → 结果写入动态层
  Phase 4 完成后（四库更新后）→ 健康检查（数据最新时）→ 结果更新
  用户开启下一章时 → 读取最新健康检查结果 → UI 展示

数据存储:
  健康检查结果不单独建表，而是作为 dynamic_layers 的 JSONB 字段存储：
  health_check: {
    checked_at: "第15章",
    alerts: [
      { rule: "R1", promise_title: "神秘人的真实身份", level: "yellow" },
      { rule: "R3", promise_title: "地宫秘密", level: "red" }
    ]
  }
```
> **为什么存到动态层？** 健康检查结果是"当前章的快照"，和动态层同一生命周期——下一章生成时会重新检查。不需要永久存储，但需要在用户打开下一章时立即显示。

### 7.1 各模式性能


| 模式           | 传统步骤     | LLM 步骤 | 预估延迟 |
| -------------- | ------------ | -------- | -------- |
| 不选·é抽 | 2            | 0        | 即时     |
| 不选·手动     | 3            | 2        | ~8s      |
| 单卡           | 6            | 3        | ~12s     |

| 双卡·冲突     | 9 + fallback | 4        | ~18s     |
| 全选           | 10           | 4        | ~20s     |
| Hybrid         | 5            | 2        | ~10s     |

> **注**：以上延迟为生成阶段（Phase 1-3）的同步延迟。收纳阶段（Phase 4 四库自动更新）在用户确认后在后台异步执行，约 +2-3s，用户无需等待即可开始下一章。

### 7.2 缓存策略


| 缓存对象     | 失效条件      | 容量         |
| ------------ | ------------- | ------------ |
| 四库过滤数据 | 选卡变化      | 每章 ~1500字 |
| 编织方案     | 选卡/权重变化 | ~200字       |
| 卡片方向文本 | 每次抽卡      | ~300字       |
| 动态层快照   | 每章版本化    | ~500字/章    |

**关键收益**: 用户调整权重滑块时，不需要重跑任何 LLM，只需要重算传统算法第①步（权重分配）→ 第⑥步（大纲填充），耗时 < 500ms。

## 九、拆书引擎（离线分析 · 冷启动数据源）

### 9.1 目标与范围

> 拆书引擎**不参与运行时生成流水线**。它是离线分析工具，产出 Genre Profile（类型知识库），用于新小说（建书）时的冷启动。

**范围限定**：只拆前 3-5 章（黄金三章 + 扩展章），不拆全本。
**理由**：墨灵只需要冷启动数据。第 4 章之后，4库 + 动态层已积累足够数据，拆书数据自然退役。

### 9.4 离线拆书算法（5 步）

#### 9.4.1 [A1] 黄金三章结构提取

```
输入: 小说前 3-5 章的原文
输出: 该小说的 beat-by-beat 结构骨架

步骤:
  a. 分段检测: 按段落分割，每段标注类型:
     - ACTION   (情节推进/冲突/动作)
     - DIALOG   (对话)
     - DESCRIPTION (环境/外貌描写)
     - EXPOSITION (设定说明/世界观输出)
     - EMOTION  (心理描写/内心独白)
  
  b. 节拍检测: 识别关键叙事节拍
     - Hook onset:      章节开头的悬念/爆点
     - Conflict onset:  冲突开始的段落
     - Revelation:      揭示信息的段落
     - Calm valley:     情绪低谷/过渡段落
     - Cliffhanger:     章尾悬念
   
  c. 构建时间轴: 将节拍映射到字数轴
     例:
     [0-200字]   Hook onset: 主角困境快照
     [200-600字] 世界渗透: 通过行为暗示世界规则
     [600-1200字] Conflict onset: 首次冲突
     [1200-1800字] Character moment: 建立角色记忆点
     [1800-2200字] Cliffhanger: 章尾钩子

  d. 去版权化: 只保留时间轴和节拍类型，丢弃原文
```
#### 9.4.2 [A2] 角色出场模式聚类

```
输入: 同一类型 N 本小说的前 3 章
输出: 角色出场时序模板

步骤:
  a. 提取每个角色的首次出场位置（章号 + 字数位置）
  b. 标注出场方式:
     - 动作优先: "他推开门，走了进去"
     - 对话优先: "'你来了。'他说。"
     - 描述优先: "一个身高八尺的男人站在门口"
     - 悬念优先: "黑暗中传来呼吸声"
  
  c. 按角色类型聚类:
     - 主角:   70% 在前 500 字内出场
     - 核心配角: 第 1 章后半段 ~ 第 2 章前半段
     - 初期对手: 第 2 ~ 3 章
     - 神秘角色: 第 3 章引入，留悬念
   
  d. 输出: 角色出场时序模板（含位置范围 + 推荐方式）
```
#### 9.4.3 [A3] 钩子密度量化

```
输入: 小说前 3 章的原文
输出: 钩子密度统计 + 钩子类型分布

钩子分类:
  - 📍 章尾悬念: 章节结束时的未解决问题
  - 🔍 信息钩:   读者知道但角色不知道/角色知道但读者不知道
  - ⏱️ 倒计时钩:  有时间压力的悬念（"距离毒发还有12小时"）
  - 💥 认知颠覆:  颠覆读者认知的转折

量化指标:
  - 平均每千字钩子数（按类型分布）
  - 钩子类型占比
  - 平均章尾钩子字数范围
```
#### 9.4.4 [A4] 节奏曲线拟合

```
输入: N 本小说的前 3 章情绪张力序列
输出: 类型标准化张力曲线

步骤:
  a. 对每章每段标张力分数 0-10
  b. 对齐字数轴（每百字取平均张力）
  c. 跨小说平均 → 标准化张力曲线

示例（玄幻 · 第一章）:
  字数段      平均张力    说明
  0-100       8.2        首段爆点（困境/冲突/悬念）
  100-300     6.5        过渡/世界渗透
  300-700     7.8        首次冲突
  700-1200    6.0        角色展示
  1200-1800   7.5        冲突升级
  1800-2000   8.8        章尾高潮
  2000-2200   9.2        章尾悬崖钩子
```
#### 9.4.5 [A5] 套路归纳 + 去版权化

```
关键规则:
  ❌ 不提取任何原文片段
  ❌ 不记录角色名/地名/专有名词
  ✅ 只提取: 节拍时序 × 字数范围 × 类型标注
  ✅ 只输出: 结构模板（JSON）+ 统计数据

去版权化门控:
  任何输出必须经过以下检查:
  1. "这能追溯到某本具体小说吗？" → 能则丢弃
  2. "这是一个通用模式还是独特创意？" → 独特则丢弃
  3. "去掉所有专有名词后还有内容吗？" → 只有占位符则丢弃
```

### 9.7 数据退役机制

```
拆书数据是"训练轮"，不是"永久数据"：

第 1 章: 拆书数据占比 80% / 用户数据占比 20%
第 2 章: 拆书数据占比 50% / 用户数据占比 50%
第 3 章: 拆书数据占比 20% / 用户数据占比 80%
第 4 章+: 拆书数据占比 0% / 用户数据占比 100%

退役触发器:
  条件 A: 4库 条目数 ≥ 15（自动触发）
  条件 B: 用户手动删除了所有预填模板（手动触发）
  条件 C: 动态层已积累 ≥ 3 章的前情（自动触发）
  条件 D: 用户写了第 5 章（强制触发）
```

## 十、连载书导入引擎（存量数据初始化）

### 10.1 目标与场景

> 用户已有 N 章连载内容（如在本地写了 50 章、从其他平台迁移过来），希望导入墨灵继续写第 N+1 章。此时 4库 和动态层都是空的，但**已有大量可用数据**。

**与建书的区别**：


|                | 建书（§9 拆书引擎）            | 连载书导入（§10 本算法）          |
| -------------- | ------------------------------- | ---------------------------------- |
| 输入           | 用户选的类型 + 一句话梗概       | 用户上传的 N 章原文                |
| 4库数据来源    | 外部类型知识库（Genre Profile） | **用户已有内容中提取**             |
| 动态层数据来源 | 类型典型开局模板                | **最近 3 章中提取**                |
| 退役           | 第 4 章后自然退役               | **不退役**（始终是用户自己的数据） |
| 角色           | 冷启动注水器                    | 存量数据转换器                     |

### 10.4 Phase 1: 全量 4库 分析算法

#### 10.3.1 分章分析策略

```
4库 是全小说跨章不变量 → 需要分析所有章节

策略: 分批 LLM 分析 + 跨章交叉验证

Step 1: 将 N 章分成若干批（每批 ≤ 15 章，避免上下文溢出）
Step 2: 每批用 1 次 LLM 调用提取该批次内的所有 4库 条目
Step 3: 跨批合并，检测同名同指 → 去重
Step 4: 跨批交叉验证，检测矛盾 → 标记
```
#### 10.3.2 [I1] 人物库提取

```
输入: 全部 N 章原文（分批次）
输出: 人物库完整数据

每条人物记录包含:
  - id:         自动生成（如 char_001）
  - name:       角色姓名
  - aliases:    别名/称号列表（如 "林霄" / "林师兄" / "霄哥"）
  - role:       角色定位（主角 / 盟友 / 对手 / 导师 / 中立 / 反派）
  - first_chapter: 首次出场章节
  - last_chapter:  最后出场章节
  - traits:     性格标签（如 ["坚毅", "重情义", "直感型"]）
  - appearance: 外貌描述
  - background: 背景故事（从各章提取合并）
  - ability:    能力描述
  - relationships: 其他角色的关系（如 {"墨导师": "师徒，林霄对其渐生产生怀疑"}）
  - status:     当前状态（"宗门考核中" / "重伤" / "失踪"...）
  - arc:        角色弧线简记（如 "从懵懂少年→发现身世→踏上寻找真相之路"）
  - chapters:   [3, 5, 7, 12, 14]  # 出场章节目录
  - updated_at: 最后更新章节

LLM Prompt 要点:
  - "提取所有有名字或明确指代的人物"
  - "同一人物的不同称呼请合并（'林师兄' = '林霄'）"
  - "只记录文中明确出现的信息，不要补充臆测"
  - "role 字段从 [主角/盟友/对手/导师/中立/反派] 中选择"
```

### 10.5 Phase 2: 近三章动态层分析

> **为什么只分析近三章？**
> 动态层是「流动事实」——只关心**故事坐标系当前状态**。分析太久远的章节会引入已过时的状态。

```
动态层 = 锚点 + 基线 + 钩子 + 变更 + 可行性

全量分析: 4库（跨章不变） ← 需要 N 章
近章快照: 动态层（当前状态） ← 只需要最后 3 章
```
#### 10.4.1 [I5] 章节锚点提取

```
输入: Chapters N-2, N-1, N
输出: 锚点数据

提取内容:
  - POV:     当前视角人物（从最后一段/最后一章判断）
  - 地点:    当前场景位置
  - 时间:    最新时间标记

方法:
  LLM 1 次调用（近三章字数有限，一次全送）
```
#### 10.4.2 [I6] 连贯性基线推导

```
输入: Chapters N-2, N-1, N
输出: must_hold 和 must_not 列表

提取逻辑:
  must_hold（必须保持）:
    - 角色当前持有的重要物品/信息
    - 已建立的重大关系变化
    - 读者已知但未揭示的真相
    - 已发生的不可逆事件
  
  must_not（必须避免）:
    - 过早揭示的秘密（如导师真实身份）
    - 违反已有设定的内容
    - 与先前章节矛盾的细节

LLM Prompt 要点:
  "从近三章中推导：哪些状态是第 N+1 章必须保持的？
   哪些是绝对不能在第 N+1 章就打破的？
   如果某件事一定要发生但不是现在，放入 must_not。"
```
#### 10.4.3 [I7] 未收束钩子识别

```
输入: Chapters N-2, N-1, N + 剧情承诺库中未回收项
输出: 钩子队列（按紧急度排序）

钩子来源:
  a. 近三章中新埋下的 → urgency=高
  b. 近三章中被推进但未回收的 → urgency=高/中
  c. 剧情承诺库中 status≠"已回收" 且最后关联章在近三章 → urgency=中
  d. 长期未触及的伏笔 → urgency=低

每条钩子记录:
  - id:        来自剧情承诺库或新生成
  - hook_text: 钩子描述
  - source_chapter: 来源章节
  - urgency:   高/中/低
  - status:    待回收/推进中

方法:
  传统算法: 从 [I3] 剧情承诺库中过滤未回收 + 近章关联 → 自动标注 urgency
  LLM 补充: 近三章中新出现的、尚未进入剧情承诺库的潜在钩子
```
#### 10.4.4 [I8] 最近变更摘要

```
输入: Chapters N-2, N-1, N vs Chapters N-5~N-3
输出: 增量变更日志

变更来源:
  - 角色状态变化（谁受了伤、谁获得新能力...）
  - 关系变化（信任→怀疑、师徒→对立...）
  - 地点切换（从某地移动到某地）
  - 新信息获得（发现秘密、得知真相...）
  - 伏笔状态变化（新埋→推进→揭示→回收）

格式:
  [章N-2] 林霄首次怀疑导师 | 获得玉佩的新线索
  [章N-1] 王玄挑衅 | 林霄压制实力应战
  [章N]   神秘人出现警告 | 林霄与墨导师对峙
```
#### 10.4.5 [I9] 可行性基线

```
输入: 综合动态层所有数据
输出: 初始置信度评分 + 待关注项

评分维度:
  - 锚点完整度: POV/地点/时间是否齐全（缺一扣分）
  - 基线完整度: must_hold 条目数 ≥ 3 则充足
  - 钩子队列: 至少有 1 个高紧急度钩子
  - 变更日志: 近三章变更是否自洽

初始置信度: 可能是 85-95%（用户自己的数据，一致性通常较好）
```

### 10.6 导入后的卡牌池生成

```
[I12] 从已有内容生成初始卡牌池

方法:
  1. 提取用户 4库 中每个角色的弧线方向 → 生成角色专属卡牌方向
  2. 提取未收束钩子 → 生成钩子推进方向
  3. 提取剧情承诺库中 midway 弧线 → 生成弧线推进方向
  4. 合并类型知识库的通用方向（补充多样性）

示例:
  用户小说 第1-50章 分析后:
    人物库 → 林霄弧线 "从懵懂到觉醒"
    → 卡牌方向: "主角在考核中初步展现真实实力"
    人物库 → 墨导师与林霄的师徒关系
    → 卡牌方向: "导师展现出更多矛盾信号"
    剧情承诺库 → 伏笔 "神秘人的真实身份"
    → 卡牌方向: "林霄发现关于神秘人的碎片线索"
```

### 10.7 边角情况处理

```
情况 A: 小说只有 1-2 章
  → 所有章节同时用于 4库 和动态层
  → 分析结果可能不完整，但导入后用户继续写即可

情况 B: 小说有 100+ 章
  → 分批分析: 每 15 章一批
  → 每批 1 次 LLM 调用
  → 100章 = 约 7 批 = 7 次 LLM 调用
  → 预计耗时: 30-60 秒（并行调用可缩短）

情况 C: 小说已有 30+ 章 → 触发 Level 2 压缩
  → 人物只保留"当前状态"字段
  → 时间线只保留近 10 章 + 重大事件
  → 与 §3.4 的四库过滤规则保持一致

情况 D: 用户从拆书建书 → 写了 5 章 → 想重新导入
  → 此时 4库已有拆书预填 + 5章自动填充
  → 直接全量分析覆盖 4库（拆书数据退役）
  → 近三章更新动态层
```

### 10.8 冲突校验 [I10]

```
4库 内部一致性检查:

1. 人物关系闭环:
   "林霄 → 墨导师（师徒）" 必须对应 "墨导师 → 林霄（师徒）"
   
2. 时间线一致性:
   事件 A 在第15章之前发生，事件 B 在第15章之后
   → 如果 A 引用 B 则报警

3. 剧情承诺闭环:
   每个"已回收"的承诺必须有对应的回收事件

4. 世界观冲突:
   "护山大阵每年重启一次" vs "护山大阵即将重启（距上次仅三个月）"
   → 标记为潜在冲突

输出: 冲突报告 + 修复建议（给用户审核）
```

### 10.9 与标准流水线的衔接

```
导入完成后:
  4库   = 完整分析结果（用户审核后）
  动态层 = 近三章分析结果（用户审核后）
  卡牌池 = 内容相关性方向 + 通用方向

进入生成流水线:
  ├── 第 N+1 章: Layer 1 动态层 ✅ 有（近三章分析结果）
  │              Layer 2 四库  ✅ 有（全量分析结果）
  │              Layer 3 卡牌  ✅ 有（内容相关 + 通用）
  ├── 第 N+2 章: 动态层更新 ← 解析第 N+1 章
  ├── ...
  └── 正常运行
```

## 十一、四库自动收纳更新（运行时 · 用户确认后触发）

### 11.1 为什么要加这个阶段？


| 之前缺了什么                      | 后果                           |
| --------------------------------- | ------------------------------ |
| 新角色登场 → 人物库不更新        | 下章 AI 忘记有这号人，角色 OOC |
| 角色伤势/心境变化 → 人物库不更新 | 下章 AI 当无事发生，逻辑断裂   |
| 新伏笔埋设 → 剧情承诺库不更新    | 伏笔不记录→忘记回收           |
| 时间推进 → 时间线库不更新        | 事件顺序混乱                   |
| 新设定揭示 → 世界观库不更新      | 世界观规则崩坏                 |

> 动态层在 Phase 3 更新了（流动事实），但四库作为**静态真理**没有被更新。这是一个架构盲区。Phase 4 补上这个缺口。

### 11.2 触发条件

```
用户确认收纳 ←──── 生成 → 用户拒稿 → 不触发
     │
     ▼
  Phase 3 完成（动态层已更新）
     │
     ▼
  Phase 4 启动
     │
     ▼
  四库更新完成 → 流入下一章
```
**关键规则：**

- **仅用户确认收纳后触发**。拒稿/废弃的章节不更新四库
- **Phase 4 在 Phase 3 之后执行**（先更新动态层，再更新四库）
- **Phase 4 更新不阻塞用户流**，可在后台异步执行（用户看到确认成功即可开始下一章）

### 11.3 Phase 4 完整状态机

> 从 confirm 请求到四库更新完成的完整状态流转。**这是 Phase 4 最核心的图——没有它，开发者分不清"入队了"和"执行了"的区别，遇到 stuck 状态更不知道怎么处理。**

```

                        │  IDLE（空闲）      │

| ------------- | ------------------- | ---------------- | ------------------ | ---------------------------------- |
| IDLE          | confirm 请求        | 幂等检查通过     | QUEUED             | 入队 celery task                   |
| QUEUED        | worker 取到任务     | —               | LOCKING            | 写入 trace_id                      |
| LOCKING       | 获取锁              | hsetnx 成功      | EXTRACTING         | 记录 locked_at/locked_by           |
| LOCKING       | 获取锁              | hsetnx 失败      | LOCK_WAIT          | 开始轮询                           |
| LOCK_WAIT     | 轮询锁释放          | 25 次内成功      | EXTRACTING         | —                                 |
| LOCK_WAIT     | 轮询超时            | 25 次后仍未释放  | FORCE_LOCK         | 强制写入锁，记录告警               |
| EXTRACTING    | LLM 调用            | 成功返回 JSON    | **VERIFYING**      | —                                 |
| EXTRACTING    | LLM 调用            | 超时/解析失败    | FAILED             | 记录错误，入回补队列               |
| **VERIFYING** | **SourceText 验证** | **全部通过**     | **WRITING**        | —                                 |
| **VERIFYING** | **SourceText 验证** | **部分失败**     | **WRITING + WARN** | **跳过失败条目，记录到 changelog** |
| **VERIFYING** | **实体名解析**      | **别名匹配完成** | **继续 WRITING**   | **名称已规范化**                   |
| WRITING       | 四库写入（事务）    | 全部成功         | DONE               | —                                 |
| WRITING       | 四库写入（事务）    | 写入异常         | FAILED             | 事务自动回滚（不再有"部分提交"）   |
| DONE          | 清理                | —               | IDLE               | 释放锁，触发卡牌淘汰               |
| FAILED        | 重试检查            | failures < 5     | QUEUED             | 自动重试                           |
| FAILED        | 重试检查            | failures >= 5    | STUCK              | 发送告警通知                       |
| STUCK         | 用户手动触发        | 点"重试"         | QUEUED             | 重置 failures 计数                 |
| STUCK         | 用户手动触发        | 点"全量重建"     | FORCE_LOCK         | 全量重新分析                       |

### 11.4 算法总览

```

**卡牌稀有度分配规则（LLM 必须严格遵循）：**

```
rarity 字段的判定标准（按优先级从高到低）:

【传说 legendary】极其罕见，平均每 5-8 章出现 1 张
  必须同时满足以下所有条件:
  - 涉及全书级别的转折（主角命运/势力格局/世界观根本改变）
  - 至少影响 3+ 个活跃中的剧情承诺
  - 具有不可逆的后果（角色死亡/永久失去/格局重写）
  - 不是阶段性进展，而是质的飞跃

【史诗 epic】重要转折
  满足以下至少一条:
  - 涉及 2 个以上核心角色的重大状态变化
  - 重大剧情转折/关键真相揭示
  - 引入全新的世界观层级/规则（非日常扩展）
  - 创建或回收一个高紧迫度剧情承诺
  - 角色关系发生根本性反转（盟友→敌对/信任→背叛）

【稀有 rare】有推进意义的进展
  满足以下至少一条:
  - 涉及 1 个核心角色的状态变化（受伤/晋级/发现）
  - 推进 1 个活跃中的剧情承诺
  - 引入新的次要角色/重要地点
  - 关联 2 个以上已有设定的新交叉点
  - 为重大转折做铺垫（伏笔升级）

【普通 common】日常推进
  符合以下任意特征:
  - 日常互动、过渡性对话、常规场景描写
  - 单一角色、无剧情转折的日常行为
  - 仅涉及世界观已有设定的平实扩展
  - 不直接影响主线剧情承诺的次要情节

质量门控:
  - 不确定时选低一级（宁低勿高）
  - 所有卡牌必须能做"一句话灵感"——如果说不清这张卡能激发什么章节方向 → 丢弃
```
**为什么要给 LLM 明确规则而不是让它自由判断？**
不同 LLM 对 "重要性" 的理解不一致。不给定规则时，同一章节内容可能输出 common/rare/epic 三种不同结果。规则约束确保输出的 rarity 具有可复现性和一致性，这是卡牌池公平性的基础。

- 确定性：程序可以直接合并数组，无需解析自然语言
- 可验证：Schema 校验失败 = 重试/降级
- 增量更新：只传变更部分，不传全库

### 11.6 数据安全保证（为什么需要验证？）

> Phase 4 是四库的"数据入口"——LLM 提取的内容直接写入四库，影响后续所有章节的生成质量。2026 年 6 月安全审计发现 **6 个数据正确性缺陷**，以下给出算法层面的保证措施。详细实现见后端设计文档 §7.12。

#### 为什么 LLM 提取会出错？


| 错误类型               | 实际场景                                     | 后果                   |
| ---------------------- | -------------------------------------------- | ---------------------- |
| **幻觉**               | LLM 提取"林夜突破筑基境"，原文根本没写       | 下一章基于错误前提生成 |
| **别名歧义**           | "林夜"vs"林夜(主角)"vs"夜儿"→ 创建 3 个角色 | 角色库膨胀、查询混乱   |
| **source_text 不匹配** | LLM 编造了 source_text 引用                  | 无法追溯来源           |
| **跨章依赖过期**       | 用户编辑第 3 章 → 第 8 章提取基于过期数据   | 数据不一致             |

#### 算法层面的四道防线

**第一道：SourceText Grounding（溯源验证）**

提取的每个条目必须包含 `source_text`（原文引用）。系统验证 `source_text` 中的内容是否真的能在原文中找到。这不是额外要求——prompt 已经有 `source_text` 字段——我们只是**执行验证**。

```
LLM 输出：{ name: "林夜", status: "筑基境", source_text: "林夜周身灵气翻涌..." }
                                  │
                                  ▼
验证：RapidFuzz 模糊匹配 source_text 与原文
  "林夜周身灵气翻涌" vs 原文中任意片段
  相似度 ≥ 85% → ✅ 通过
  相似度 < 85% → ⚠️ 可疑 → LLM-as-Judge 二次确认
                    → FAIL → 跳过该条目 + 记录 changelog
```
**为什么用模糊匹配？** LLM 输出的 `source_text` 可能与原文字面有微小差异（标点、省略修饰语）。精确匹配会误杀有效内容。RapidFuzz 的 `partial_ratio` 算法会忽略前后缀差异，适合中文文本。

**第二道：实体名规范化（别名去重）**

每次 Phase 4 提取时，对出现的角色名执行 RapidFuzz 模糊匹配：

```
提取到 "林夜(主角)" → 匹配已有实体 "林夜"（相似度 92% ≥ 85%）
  → 自动注册别名："林夜(主角)" → "林夜"
  → 该条目的变更直接合并到 "林夜" 名下
  → 下次提取 "林夜(主角)" 时，精确匹配直接返回 "林夜"（O(1)）

提取到 "新角色张三" → 无匹配（相似度 < 85%）
  → 注册为新实体
  → 后续提取的别名会逐渐积累到 "张三" 名下
```
**为什么这是算法问题而不是工程问题？** 实体解析发生在数据入库之前。如果在工程层（写入时）才处理，已经产生了脏数据。在算法层处理意味着**脏数据永远不会产生**。

**第三道：事务原子写入**

四库的 6 个子库更新（人物/时间线/剧情承诺/世界观/卡牌/秘密）+ 变更日志归档，必须在**同一个数据库事务**中完成：

```
旧行为：逐库写入 → 第 3 步失败 → 前 2 步已提交 → 部分数据脏
新行为：单事务写入 → 任何一步失败 → 全量回滚 → 数据一致

为什么？算法层面的"部分更新"比"完全失败"更危险。
完全失败 → 用户知道有问题，可以重试
部分更新 → 用户看到"收纳成功"，但四库只有一半数据 → 信任崩塌
```
**第四道：跨章依赖追踪**

当用户编辑第 N 章原文时，后续第 M 章（M > N）的 Phase 4 提取可能基于过期的四库数据：

```
用户编辑第 5 章 → 系统追踪：
  "伏笔: 神秘人身世" 基于第 3 章提取 → 第 7 章、第 12 章引用了这个伏笔
  "角色: 林夜" 基于第 5 章提取 → 第 6 章、第 8 章引用了这个角色

结果：标记第 7、12 章为 stale（过时），下次用户查看时提示"建议重新提取"
```
**为什么是"惰性重算"而不是"自动重算"？**

- 自动重算 10 章 = 10 次 LLM 调用 = 成本
- 用户可能只关心第 7 章的结果，不需要重算第 12 章
- 惰性重算 = 用户需要时再算 = 按需付费

    ② 检查 name 是否与已有角色模糊匹配（编辑距离 < 3）:
       - 匹配 → 降级为 update（或标记为用户确认）
       - 不匹配 → 创建新角色，初始状态从 new_entries 填充
    ③ 设置登场章节 = 当前章号

  action=update:
    ① 用 target_id 或 name 查找已有角色
    ② 对每个 changes[].field:
       - field 已存在且值不同 → 记录变更历史，更新为新值
       - field 不存在 → 新增字段
       - old 值在动态层中标记为"历史状态"
    ③ evidence 存入该角色的"变更溯源"字段

  action=status_change:
    ① 更新角色的"当前状态"字段
    ② 旧状态存入"历史状态"栈（保留最长 30 章）
    ③ 如果状态涉及"重伤/死亡/背叛"等重要转变 → 自动添加剧情承诺提醒

  action=remove:
    ① 不移除角色数据（保留以防回溯）
    ② 标记状态为"退场"，设置退场章节
    ③ 检查是否有该角色的未收束承诺 → 标记为"角色退场·承诺悬空"

  confidence < 0.5:
    ① 不自动执行，将建议加入用户审查队列

**模糊匹配算法（保护已有数据不被覆盖）：**

```
function findCharacter(name, existingChars):
  // 1. 完全匹配
  if existingChars[name] return existingChars[name]
  
  // 2. 别名匹配（主角·苏铭 / 苏铭）
  for char in existingChars:
    if name in char.aliases return char
  
  // 3. 编辑距离 ≤ 2（处理 OCR/LLM 输出误差）
  for char in existingChars:
    if levenshtein(name, char.name) <= 2 return char
  
  // 4. 姓氏+首特征匹配（"苏少" → "苏铭"）
  for char in existingChars:
    if name[0] == char.name[0] and 
       (name.includes(char.short_desc[0]) or 
        char.short_desc.includes(name[0:2])) return char
  
  // 5. 无匹配 → 新角色
  return null
```

| 指标           | 数据                                |

| 小模型         | 足够（仅需信息提取，无需生成）      |
| 预估延迟       | ~2-3s（异步，不阻塞用户流）         |
| 输出数据大小   | ~1-3KB JSON                         |

| 误判率         | 预估 < 5%（结构化约束大幅降低幻觉） |

| 全库重新分析   | 按需触发（用户主动要求时）          |

### 11.12 为什么只做增量不做全量？


| 方案                   | 优点                         | 缺点                           | 为什么选增量            |
| ---------------------- | ---------------------------- | ------------------------------ | ----------------------- |
| **增量提取**（本方案） | 1 次 LLM 调用/章，~2-3s 异步 | 可能遗漏跨章关联               | ✅ 成本低、速度快、够用 |
| 全量重分析 N 章        | 完整覆盖                     | N 次 LLM 调用/章，成本线性增长 | ❌ 10 章后不可接受      |
| 每 10 章全量重分析     | 定期校准                     | 架构复杂、增量+全量双通道      | ❌ 过度设计             |
| 仅靠规则引擎           | 0 LLM 成本                   | 无法理解叙事内容，覆盖极有限   | ❌ 不可行               |

### 11.13 长篇记忆增强：实体关系图

> **为什么需要这个？**
>
> 增量提取"只处理æ¬章新内容，不主动回顾历史"的设计，在 10 章以内够用，但 50 章后会出现跨章关联遗漏。调研发现，这其实是 **LLM Agent 长期记忆** 领域的共有挑战，业界已有成熟方案。

#### 11.13.1 核心洞察：三个问题，一个解法


| 审计指出的问题      | 根因                                 | 统一解法                                               |
| ------------------- | ------------------------------------ | ------------------------------------------------------ |
| 长篇记忆可靠性低    | 四库传"压缩摘要"而非实体间的关系网络 | **实体关系图**：在四库之上增加关系层，支持多跳检索     |
| 正史防污染能力低    | 新提取 vs 已有数据不做对比           | **一致性校验**：写入前对比新旧数据，检测矛盾           |
| 伏笔/秘密控制能力低 | 只存 JSONB 字段，不存"谁跟谁有关"    | **关系追踪**：伏笔/秘密作为图节点，与角色/事件显式关联 |

**为什么是"关系图"而不是更复杂的方案？**


| 方案                     | 评估                                               |
| ------------------------ | -------------------------------------------------- |
| 全量重算历史             | 0 额外架构但成本线性增长，50 章后不可接受          |
| 增大 LLM 上下文窗口      | 32K→128K 是可行的，但 128K 后继续增长？无限膨胀？ |
| **实体关系图**           | 一次提取，永久检索。成本上限可控，不随章节数增长   |
| 向量数据库（纯语义搜索） | 能找"相似"但找不出"谁知道什么"的关系               |
| 独立图数据库（Neo4j）    | 强大但运维重，4C4G 服务器负担大                    |

**选择：在 PostgreSQL 中建轻量关系表，与现有四库 JSONB 互补。**

#### 11.13.2 算法原理

```
传统方案（仅 JSONB）：
  四库存储 → 压缩后传给 LLM → LLM 凭记忆关联

增强方案（JSONB + 关系图）：
  四库存储 + 关系图存储
              ↓
  从本章角色出发，在关系图中做 2-hop 遍历
              ↓
  找到相关历史实体（伏笔/秘密/事件）
              ↓
  作为额外上下文注入 LLM Prompt
              ↓
  LLM 看到的不只是当前章节，还有"与本章角色相关的历史脉络"
```
**2-hop 遍历示例：**

```
当前章节提到角色「林夜」

1-hop（直接关系）:
  林夜 → 知道 → 秘密「玉佩是钥匙」
  林夜 → 参与 → 事件「秘境探索」
  林夜 → 关联 → 伏笔「身世之谜」

2-hop（间接关系）:
  林夜 → 知道 → 秘密「玉佩是钥匙」← 不知 ← 苏晚晴
  林夜 → 参与 → 事件「秘境探索」← 参与 ← 叶灵
  林夜 → 关联 → 伏笔「身世之谜」← 关联 ← 墨导师

结果：LLM 不仅知道林夜在第 50 章做了什么，
      还知道苏晚晴不知道的秘密、叶灵参与过的事件。
      这显著提升长篇叙事的连贯性。
```
#### 11.13.3 一致性校验算法

```
写入前校验（Detect-Then-Resolve 框架）:

Phase 1 — Detect（检测）:
  
  确定性规则（纯代码，零 LLM 成本）:
    ├── 角色状态互斥：同时 alive 和 dead？→ 矛盾
    ├── 年龄单调：从 20 岁变回 18 岁（无回溯解释）？→ 矛盾
    ├── 位置连续性：瞬间从北域到南疆（无传送能力）？→ 矛盾
    └── 秘密降级：已公开的秘密又变回绝密？→ 矛盾
  
  语义规则（按需 LLM 调用）:
    └── 修为变化：金丹→筑基（合理，可能是受伤）
                   金丹→筑基→元婴（波动太大，需要确认）

Phase 2 — Resolve（解决）:
  
  高严重度矛盾（状态互斥、年龄回退等）:
    └── 阻止自动写入 → 暂存 staging → 等待用户确认
  
  中低严重度矛盾（修为波动等）:
    └── 记录到 changelog → 允许写入 → 用户可在四库管理页查看

为什么分两级？
- 确定性规则：O(1) 检查，覆盖 80% 的矛盾场景
- 语义规则：按需触发，仅 ~10% 的字段变更需要
- 总成本几乎为零
```
#### 11.13.4 伏笔/秘密增强追踪

```
传统能力（已有）:
  - 伏笔有 type/status/event_type 字段
  - 秘密有 secrecy_level/known_by/unknown_to 字段
  
增强能力（关系图）:
  - 伏笔是图节点 → 通过边关联到角色、事件、秘密
  - 秘密是图节点 → 通过边关联到所有知情/不知情角色

新增的可回答问题:
  - "主角知道的秘密中，哪些是反派还不知道的？"
    → 2-hop: 主角→KNOWS→秘密←NOT_KNOWN_BY←反派
  - "这个伏笔被哪些事件推进过？"
    → 直接查: 伏笔→ADVANCED_BY→事件1, 事件2, ...
  - "哪些伏笔超过 15 章未推进？"
    → SQL: WHERE node_type='promise' AND is_active=TRUE AND 
           (chapter - last_seen_chapter) > 15
```

## 十二、Phase 4 调度器（运行时引擎）

> Phase 4 不是"跑完就算"的脚本，而是**需要在生产环境中可靠执行的并发模块**。本节定义调度器的数据结构、竞态防护、失败恢复和幂等性保证。

### 12.1 调度器状态数据结构

```javascript
Phase4Scheduler = {
  // ── 执行队列 ──
  queue: [
    // { chapter: 16, nonce: "ch16_1712345678", status: "pending"|"running"|"done"|"failed" }
  ],
  
  // ── 幂等性检查：已执行的 nonce（避免重复执行）──
  executedNonces: Set<string>,
  
  // ── 四库版本锁 ──
  vaultVersion: {
    current:     "v4_ch15_1712345000",  // 四库当前版本（最新成功写入）
    pending:     null,                   // 正在写入的版本号（写入中非空）
    lastStable:  "v4_ch15_1712345000",  // 最后确认成功的版本（回滚用）
    writeLock:   false                   // 写入锁（true 时禁止并发写）
  },
  
  // ── 失败回补 ──
  fallbackQueue: [],     // [14, 15] Phase 4 失败的章号（按序）
  consecutiveFailures: 0 // 连续失败计数
}
```

### 12.3 竞态防护：下一章 Phase 1 门控

**问题**：Phase 4 是异步的（~2-3s），但用户确认后可以立刻开始下一章。如果 Phase 4 还没写完四库，下一章的 Phase 1（四库过滤）会读到过期数据，甚至出现读写并发。

**解决方案：乐观等待 + 降级读取**

```
用户在章 N 点击确认收纳:
  → UI 显示"收纳成功" ✅（用户体验不阻塞）
  → Phase 4 启动（后台）

用户点击"开始第 N+1 章":
  → 门控检查:
      if vaultVersion.pending !== null:
        // 四库正在写入
        showToast("四库收纳中，请稍候…")
        await waitForVaultReady(5000)  // 最多等 5 秒
          .then(()  → 使用最新四库）
          .catch(() →
            // 超时了，但不想让用户干等
            // 降级策略：使用 lastStable 版本四库
            // 并标记"四库数据可能存在微小延迟"
            loadChapterWithVault(lastStable)
          )
      else:
        → 直接使用 current 版本四库
```
**读写锁保护**（防止并发写入）：

```
当 Phase 4 正在写入四库时:
  任何并发的 Phase 4 写入请求 → 等待 200ms 重试
  如果锁占用超过 10 秒 → 强制解锁 + 标记异常

例外：全量重新分析（用户手动触发）时:
  等待所有 pending Phase 4 完成后才执行
  期间禁止新 Phase 4 入队
```
### 12.4 失败回补策略

**核心原则**：四库最终必须与正文一致。一两个章节的遗漏不应影响用户继续创作，但也不能永久沉默。


| 失败次数     | 处理                                                                          | 用户可见性 |
| ------------ | ----------------------------------------------------------------------------- | ---------- |
| 1 次（偶发） | 自动合并到下一章 Phase 4（§12.2 Step 3 fallback_merge）                      | 不可见     |
| 连续 2 次    | 同一策略，但四库管理页出现 "2 章待补" 角标                                    | 低可见性   |
| 连续 3 次    | 弹窗："检测到 {N} 章的四库变更尚未收录。建议全量重新分析。 [立即分析] [稍后]" | 中等       |
| 连续 5 次    | 强制暂停生文，必须先解决四库问题                                              | 高         |

**回补合并示例**：

```
第 16 章 Phase 4 失败 → fallbackQueue = [16]
  用户写第 17 章 → 确认收纳
  第 17 章的 Phase 4 Prompt 自动追加:
    "⚠️ 注意：第 16 章的四库变更也尚未收录。
     请在分析第 17 章的同时，对第 16 章中可能出现的
     角色/事件/伏笔变更一并提取。"
  合并输出 → 17 章的 Phase 4 同时补上 16+17 两章的增量
  → 16 章从 fallbackQueue 移除
```
### 12.5 幂等性保证

**为什么需要三层防护？**

Phase 4 是异步执行的，网络抖动、用户双击、Redis 重启都可能导致同一请求被多次执行。如果每次执行都产生新数据，四库会被重复条目污染。


| 场景                                     | 防护机制                                 |
| ---------------------------------------- | ---------------------------------------- |
| 用户快速点击两次"确认收纳"               | 第一次触发后，UI 按钮置灰 disabled       |
| 网络超时 → 用户重试 → 第一次其实成功了 | Layer 1: Redis`sismember` 快速检查 nonce |
| Redis 重启 → nonce 丢失 → 同一请求重入 | Layer 2: DB 查询`chapter_phase4.nonce`   |
| 应用程序层和前两层同时失效               | Layer 3: DB`UNIQUE` 约束（永不失效）     |

**三层防护架构**：

```
                    HTTP 请求（含 nonce）
                           │
                           ▼
              ┌───────────────────────┐
              │ Layer 1: Redis 检查    │ ← ~1ms，覆盖 99.9%
              │ sismember(nonces, nonce)│
              └────────┬──────────────┘
                  已存在│               │不存在
                       ▼               ▼
                    return <跳过>  ┌───────────────────────┐
                                  │ Layer 2: DB 检查       │ ← ~5ms，Redis 重启保护
                                  │ SELECT nonce FROM      │
                                  │ chapter_phase4 WHERE   │
                                  │ nonce = ?              │
                                  └────────┬──────────────┘
                                      已存在│               │不存在
                                           ▼               ▼
                                    Redis 恢复     ┌────────────────────┐
                                    nonce          │ 执行业务逻辑        │
                                    return <跳过>   │ 写入时自动写入 nonce │
                                                    └────────┬───────────┘
                                                             │
                                                    ┌────────▼──────────┐
                                                    │ Layer 3: DB UNIQUE │ ← 永不失效
                                                    │ 约束自动防重       │
                                                    │ IntegrityError →   │
                                                    │ return <已存在>    │
                                                    └───────────────────┘
```
**为什么需要 DB 层？**

- Redis 是易失存储，重启后 nonce Set 丢失
- 但 Phase 4 写入是持久操作，幂等性必须是持久保证
- DB 的 UNIQUE 约束不依赖任何中间件状态
- 这是"最终防线"——前两层失效时，DB 仍然拒绝重复

**nonce 生成规则**：

```
nonce = `ch${chapter}_${Date.now()}`
  例: "ch16_1712345678"

同一章的两次 nonce 不同（时间戳不同）→
  第一次写入成功后，executedNonces 记录该 nonce
  第二次遇到不同 nonce 但同一章号 →
    检查是否已经有一份同一章的变更写入成功 →
    有则跳过（幂等兜底）
```
### 12.6 和现有流程的关系

Phase 4 调度器**不改变** §11.7-11.12 的各库合并逻辑。它只负责：

1. **触发**：在 `confirmDraft()` 被调用时启动 Phase 4
2. **验证**：在写入前执行 SourceText 验证和实体名规范化
3. **排队**：确保同一时间只有一个 Phase 4 在写四库
4. **事务**：保证所有写入在同一个 DB 事务中，失败全量回滚
5. **重试**：网络抖动时自动重试 1 次
6. **回补**：失败时将遗漏章节加入回补队列
7. **门控**：下一章开始时检查四库是否可用
8. **幂等防护**：Redis + DB 双层 nonce 检查

**§12.10 的兜底规则用本调度器替代**（旧规则过于简单，未覆盖竞态和幂等场景）。
