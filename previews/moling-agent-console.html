设计决策说明

决策维度 传统 SaaS 做法 墨灵 Agent Console 做法 契合的产品特点
主交互模式 表单 + 按钮驱动 对话 + 自然语言驱动 写作是创意活动，用户想说"帮我写这个"而非填表
操作入口 顶栏工具栏散落各处 快捷指令条(Quick Actions) + 命令式输入 Agent 应像伙伴：告诉它做什么，不需要找按钮
上下文管理 切换页面查看不同数据 四库持久侧栏，Agent 自动引用 四库是长篇小说的"记忆系统"，必须始终可见
输出方式 原地替换编辑器内容 右侧预览面板 + 采纳/编辑/重生成 AI 输出不是最终版，用户需要"审查-采纳-修改"循环
抽卡集成 独立弹窗或页面 嵌入对话流中，AI 主动提议抽卡 抽卡是创作启动器，应在对话中自然出现
状态反馈 Loading spinner Agent 思考过程可见(Thinking Steps) 让用户看到 AI 在读什么数据、为什么这样输出
导航策略 多页面跳转 单页面三栏：四库 + 对话 + 输出 创作是沉浸式活动，减少页面跳转 = 减少心流打断

为什么不做成 ChatGPT 式的纯对话

纯对话界面对网文创作有问题：① 四库数据(人物/世界观/情节)需要持久可查，不能靠"翻聊天记录"；② 输出内容需要编辑和版本管理；③ 抽卡机制是墨灵的差异化特色，必须视觉化呈现。

所以方案是 "对话驱动 + 三栏布局"：左边的四库和右边的输出是 Agent 的"双手"，中间的对话是 Agent 的"大脑"。

快捷指令设计哲学

底部快捷指令(续写/重新抽卡/大纲优化/伏笔检查/语气校准)不是按钮，是 语境的提示。 每个都对应一个高频操作，点击后填充输入框并自动发送——用户不需要记住命令，看到就知道能做什么。这 5 个指令的选取是基于"网文创作的最核心五步"：

续写当前章节
最高频操作，放在首位
重新抽卡
不满意就重来，墨灵核心机制
大纲优化
结构层操作，章节级别
检查伏笔一致性
长篇创作的最大痛点
角色语气校准
