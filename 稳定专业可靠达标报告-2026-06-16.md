## 五、关键经验与 Learning 记录

### 1. 多 Agent 并行加速大型任务
- **经验**: 8 个 Agent 并行执行，总耗时 ~2 小时（串行需要 ~30 小时，加速 15x）
- **要点**: 
  - 按任务类别分组（P0/P1/P2）
  - 确保 Agent 之间无文件冲突
  - 先用小任务验证 Agent 能力，再派发大任务
- **记录到**: `.learnings/LEARNINGS.md`（新增 LRN-20260616-001）

### 2. API 对齐必须自动化
- **经验**: 手动对比 50 个端点容易遗漏，应该用 OpenAPI schema 自动生成契约文档
- **要点**:
  - FastAPI 自动生成 OpenAPI schema（`/openapi.json`）
  - 写脚本对比 `api.ts` 中的调用路径
  - 在 CI 中添加 API 契约检查步骤
- **记录到**: `.learnings/LEARNINGS.md`（新增 LRN-20260616-002）

### 3. 安全加固的 5 个必需品
- **经验**: Rate Limiting、JWT 黑名单、HTTPS、SQL 注入防护、CSP 是"专业级"应用的最低要求
- **要点**:
  - Rate Limiting 用 `slowapi`（FastAPI 生态）
  - JWT 黑名单用 Redis（轻量级，低延迟）
  - HTTPS 用 Let's Encrypt（免费，自动化）
- **记录到**: `.learnings/LEARNINGS.md`（新增 LRN-20260616-003）

### 4. 性能压测应该在开发早期执行
- **经验**: 如果等到上线前才压测，发现性能瓶颈后修改成本高
- **要点**:
  - 在 MVP 阶段就建立性能基线
  - 每次重大更新后重新压测
  - 在 CI 中添加性能回归检测
- **记录到**: `.learnings/LEARNINGS.md`（新增 LRN-20260616-004）

### 5. 运维文档是"专业级"的标志
- **经验**: 有完整 RUNBOOK 的项目，故障恢复时间缩短 80%
- **要点**:
  - RUNBOOK 要详细（步骤级，不是概述）
  - 定期演练（每季度一次）
  - 每次故障后更新 RUNBOOK
- **记录到**: `.learnings/LEARNINGS.md`（新增 LRN-20260616-005）
