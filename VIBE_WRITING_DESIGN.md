# Vibe Writing — 革命性网文写作产品设计文档

## 一、设计哲学：从 Codex 到 Vibe Writing

### Codex 对编程行业的颠覆性创新

| Codex 原则 | 编程领域实现 | Vibe Writing 映射 |
|-----------|-------------|------------------|
| Agent of Agents | 多专业 Agent 协作编码 | 剧情/人物/对话/风格/世界观 Agent 协作写作 |
| Intent > Manual Labor | 描述意图，Agent 生成代码 | 选择方向，Agent 生成内容 |
| Heavy Options, Light Input | 预设选项推进，减少手动输入 | A/B/C 选项 + 最小化 D 自由输入 |
| Context is King | 深度理解项目上下文 | 深度理解小说状态、阶段、风格 |
| Flow State | 不打乱开发者心流 | 不打乱写作者创作流 |
| Parallel Execution | 并行多任务处理 | 并行多维度写作推进 |
| Spec-Driven | 先规格后实现 | 先构思后写作 |

### Vibe Writing 核心原则

1. **选择优于输入 (Choose > Type)** — 80% 操作是选 A/B/C，20% 是必要自由输入
2. **代理协作 (Agent Collaboration)** — 多个 AI Agent 各司其职，协调产出
3. **上下文自适应 (Context-Adaptive)** — 系统深度理解当前小说阶段，智能生成选项
4. **心流保护 (Flow Protection)** — 绝不打断创作者思路，无缝衔接上下文
5. **渐进式展开 (Progressive Disclosure)** — 从宏观到微观，逐步细化

### 交互流

```
1. 用户进入写作阶段
2. Coordinator 分析上下文，调用相关 Agent
3. 各 Agent 基于小说状态生成提案
4. Coordinator 整合提炼为 A/B/C 选项 + 预览
5. 用户选择或输入
6. 所选 Agent 执行，更新小说状态
7. 循环 → 回到步骤 2
```

## 六、产品差异化对比

| 维度 | 传统写作工具 | AI 写作助手 | Vibe Writing |

## 八、前端交互实施决策

> 2026-06-20 | 基于实际开发中确定的交互行为规范

### 8.1 Sidebar — 小说项目导航

```
┌──────────────────┐
│  新建项目 / 小说   │  ← 入口
├──────────────────┤
│  ▼ 剑道巅峰       │  ← 活跃项目（展开）
│   第3章 剑指苍穹  │
│   第2章 突破       │
│   第1章 初入宗门   │  ← 倒序，最新章紧贴项目名
│  ▸ 星际流浪者     │  ← 非活跃（折叠）
├──────────────────┤
│  知识中心          │  ← 占位
│  插件市场          │  ← 占位
├──────────────────┤
│  用户设置          │
└──────────────────┘
```

**交互规则**：

| 规则 | 说明 |
|------|------|
| 点击小说名 | setActiveProject + toggleExpand + navigate |
| 展开 = 切换活跃项目 | 同一时间只有一个项目展开 |
| 章节倒序 | 最新章（ID 最大）紧贴项目名下方 |
| 非活跃项目章节 | disabled + dimmed，不可点击 |
| 展开项目 ≠ 活跃项目 | 不可点，视觉上告知用户需先切活跃 |

**连载中 vs 已完结分栏**：

| 状态 | 判断条件 | 行为 |
|------|----------|------|
| 连载中 | 存在未完成章节 | 可正常交互 |
| 已完结 | 全部 chapter.status === "completed" | 侧栏灰色「完结」徽章 · Workspace 无 OptionsPanel · 只读 |

### 8.2 章节新增 — 自动创建

```
完成本章 → 标记 completed → 自动创建 Chapter N+1 (draft, 空内容) → 留在当前章
```

| 决策 | 原因 |
|------|------|
| **不跳转到新章** | 用户可能需要回顾刚完成的章节，不应强制跳转 |
| **自动创建而非手动按钮** | 减少操作摩擦，保持写作心流 |
| **新章默认 draft** | 空内容，等待用户点击进入后才开始填充 |

`completeChapter` 逻辑：

```
if 所有计划章节已完成:
  → 项目进入「已完结」，phase = revision
else:
  → 当前章 status = "completed"
  → 创建 nextChapter (id+1, draft, 空内容)
  → currentChapter 保持为刚完成的章（不跳转）
  → 新章出现在侧栏章节列表底部（倒序渲染后在最前）
```

### 8.3 状态条 — 顶部左对齐 + 三态业务驱动

**位置**：顶部栏最左侧，与主题切换/右栏按钮并列。

```
| ✏️ 写作中 · 草稿 |  ---spacer---  |  🎨 主题  |  📋  |
```

**三态逻辑**：

| 状态 | 触发条件 | 图标 | 文案 | 颜色 |
|------|----------|------|------|------|
| 已完结 | 全部章节 completed | CheckCircle | `已完结` | accent |
| 写作中 | 当前章 = 末章且非 completed | Edit3 | `写作中 · {阶段名}` | accent |
| 回顾中 | 连载中但查看往期已完成章节 | Eye | `回顾中` | muted |

**关键修复**：`lastChapterId` 不再依赖 `chapters.length`（脆弱），改用 `chapters[chapters.length - 1]?.id`（以实际数据为准）。

### 8.5 面板缩放 — 左右栏可拖拽

**实现**：原生 mouse 事件拖拽，无第三方依赖。

| 面板 | 默认宽度 | 最小 | 最大 | 备注 |
|------|----------|------|------|------|
| 左栏（Sidebar） | 240px | 160px | 400px | 折叠态固定 44px，不可拖拽 |
| 右栏（AgentPanel） | 260px | 200px | 500px | 关闭后不渲染 |

**交互**：
- 拖拽手柄 6px 宽，默认透明（`opacity-0`），hover 时半透明显示品牌色
- 手柄两侧各有 4px 不可见热区（`-left-1 -right-1`），降低操作精度要求
- 拖拽时 `body` cursor 变为 `col-resize`，松开复原
