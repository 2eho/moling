## 一、项目背景与目标

### 1.1 行业背景

#### 1.1.1 网文市场现状

中国网络文学市场规模已突破 **300 亿元/年**，创作者数量超过 **2000 万**，但：
- **95% 的创作者** 面临"卡文"困境（写不下去）
- **80% 的弃书** 发生在前 30 章（开篇失败）
- **传统写作助手** 仅提供"扩写/改写"，无法解决结构性问题

#### 1.1.2 核心痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| **卡文** | 不知道接下来发生什么 | 创作中断，弃书 |
| **人物 OOC** | 写着写着人物性格变了 | 读者流失 |
| **逻辑漏洞** | 前面埋的伏笔忘了回收 | 剧情崩塌 |
| **风格失控** | 搞笑文写着写着变严肃 | 类型偏移 |
| **开篇失败** | 前三章留不住读者 | 无法签约 |

#### 1.1.3 现有解决方案的局限

| 产品 | 方案 | 局限 |
|------|------|------|
| **Jasper/JCopy** | 续写 + 改写 | 无故事记忆，越写越偏 |
| **NovelAI** | 纯 LLM 生成 | 失控（OOC、逻辑漏洞），无法干预 |
| **Scrivener** | 写作项目管理 | 无 AI 辅助，纯工具 |
| **Dramatron** | 剧本结构生成 | 学术研究，无产品化 |

**市场空白**：缺少一个**既能保持创作者控制权，又能解决卡文问题**的 AI 写作助手。

---

### 1.2 产品定位

**墨灵**是一款 **AI 辅助小说创作平台**，核心创新是：

```
抽卡获取灵感 → 组合卡片生成章节 → 四库维护故事一致性
```

**核心差异**：
- ❌ **不是** "AI 帮你写"（NovelAI）
- ✅ **而是** "AI 给你灵感，你来决定怎么写"（Hybrid Pipeline）

**目标用户**：
1. **网文作者**（起点/晋江/番茄）→ 解决卡文、提高产出
2. **编剧/小说爱好者** → 降低创作门槛
3. **写作教育工作者** → 教学辅助工具

## 三、实现方案与关键技术

### 3.1 卡牌组合生文算法（核心创新）

#### 3.1.1 问题定义

**输入**：
- 用户选择的卡牌集合（3-5 张）
- 当前章节的动态层状态
- 四库（人物/时间线/剧情承诺/世界观）

**输出**：
- 章节正文（Markdown）
- 更新后的动态层
- 四库变更建议（Phase 4）

**约束**：
- 保持人物性格一致（OOC 检测）
- 保持文风一致（风格指纹）
- 保持逻辑连贯（伏笔回收）

#### 3.1.3 关键技术点

| 步骤 | 技术 | 实现方式 |
|------|------|---------|
| **[1] 权重分配** | 卡牌优先级算法 | 基于卡牌类型、新鲜度、用户偏好的加权排序 |
| **[2] 四库过滤** | 向量检索 + 规则过滤 | pgvector 语义搜索 + 人物/时间线约束 |
| **[3] 动态层冲突检测** | 规则引擎 | 检测人物状态、位置关系、时间线矛盾 |
| **[4] 方向冲突评分** | 多维度评分 | 情节方向、人物弧光、伏笔回收的冲突度 |
| **[5] 编织方案匹配** | 模式匹配 | 预定义编织模式（暗恋+误会、复仇+反转等） |
| **[6] 大纲模板填充** | 模板引擎 | 基于卡牌组合的动态调整模板 |
| **[7] 叙事元素提取** | LLM Prompt | 从卡牌组合中提取核心叙事元素 |
| **[8] 头脑风暴** | LLM Chain-of-Thought | 多轮对话生成情节变体 |
| **[9] 正文写作** | LLM Streaming | 流式输出，实时渲染 |
| **[10] 连贯性校验** | 规则 + LLM | 检测 OOC、逻辑漏洞、风格偏移 |
| **[11] 动态层更新** | Diff 算法 | 提取章节中的状态变更 |
| **[12] 前情摘要更新** | LLM 摘要 | 生成读者需要了解的背景信息 |

#### 3.5.2 调度策略

|------|------|

| **Rate Limit 规避** | 检测 429 错误，自动切换到下一个 Key |

### 3.6 卡牌生命周期

#### 3.6.1 状态机

```
┌──────────┐
│  Fresh   │ ← 新建/抽中
│  (新鲜期) │
└────┬─────┘
     │ 使用 3 次后
     ▼
┌──────────┐
│ Retired  │ ← 可用的，但优先级降低
│  (退役)   │
└────┬─────┘
     │ 超过 30 天未使用
     ▼
┌──────────┐
│ Eliminated│ ← 归档，不再出现在抽卡池
│  (淘汰)   │
└──────────┘
```

---

#### 3.6.2 新鲜期保护

- **规则**：新抽到的卡牌，前 3 次使用优先级 × 2
- **目的**：鼓励用户尝试新卡牌，避免总是用老卡牌
- **实现**：`DrawStateMachine` 状态机

### 3.7 健康监控系统

#### 3.7.1 三色规则

| 规则 | 检测内容 | 阈值 | 动作 |
|------|---------|------|------|
| **R1** | 子情节健康度 | < 30% | 提示用户"某某情节线快消失了" |
| **R2** | 伏笔未回收 | > 10 章 | 提示用户"某某伏笔还未回收" |
| **R3** | 人物弧光停滞 | > 5 章无变化 | 提示用户"某某角色最近没变化" |

---

#### 3.7.2 实现方式

```
每章节生成后 → 健康检查器运行
    ↓
[R1] 扫描所有子情节 → 计算进度百分比
    ↓
[R2] 扫描所有剧情承诺 → 计算未回收章节数
    ↓
[R3] 扫描所有人物 → 计算状态变更频率
    ↓
生成健康报告 → 显示在 Workspace 右侧
```

## 六、项目亮点与创新点

### 6.1 核心创新

#### 6.1.1 混合流水线架构（Hybrid Pipeline）

**创新点**：
- 传统算法（确定性） + LLM 生成（创造性）的结合
- 前 6 步使用传统算法（可缓存、低成本）
- 后 3 步使用 LLM（高质量生成）
- **成本优化**：相比纯 LLM 方案，Token 消耗降低 **70%**

**技术实现**：
```
传统算法（6 步） → 结构化蓝图（JSON） → LLM 生成（3 步） → 后处理（3 步）
```

---

#### 6.1.2 四库 + 动态层架构

**创新点**：
- **四库**（静态事实）：跨章节不变，维护故事一致性
- **动态层**（流动事实）：每章节更新，追踪故事状态
- **Phase 4 收纳**：异步更新四库，无需用户手动维护

**解决问题**：
- ❌ 其他产品：LLM 生成越写越偏（无记忆）
- ✅ 墨灵：四库 + 动态层保证一致性

---

#### 6.1.3 卡牌组合生文

**创新点**：
- 将"灵感"具象化为**卡牌**
- 用户通过**抽卡**获取灵感（游戏化体验）
- 卡牌组合 → 算法 → 生成章节
- **新鲜期保护**：鼓励尝试新卡牌

**用户体验**：
```
卡住不知道写什么 → 抽卡 → 看到灵感提示 → 选择卡牌组合 → 生成章节
```

---

#### 6.1.4 风格指纹系统

**创新点**：
- 7 维文风分析（句长、对话占比、标点密度等）
- 将"文风"量化为可计算的特征向量
- 生成时注入风格 Prompt，保持文风一致

**解决问题**：
- ❌ 其他产品：生成内容风格漂移
- ✅ 墨灵：风格保持度 > 70%

---

### 6.2 技术亮点

#### 6.2.1 异步原生架构

- **FastAPI** 原生支持异步（async/await）
- **SQLAlchemy 2.0** 异步 ORM
- **Celery** 分布式异步任务
- **流式输出**：LLM 生成时实时渲染（无需等待全部完成）

---

#### 6.2.2 向量检索（pgvector）

- 无需独立部署向量数据库（如 Pinecone）
- 与业务数据同源（事务支持）
- 人物/情节相似度搜索
