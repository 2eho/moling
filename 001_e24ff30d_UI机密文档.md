# 墨灵 · UI 机密文档

> 本文档记录 UI 中哪些信息涉及算法知识产权，需对用户隐藏/模糊化。
> **原则**：让用户知道「能用」，不让对手知道「怎么做」。

---

## 一、核心保护逻辑

墨灵的竞争壁垒在于**三层注入架构 + 混合流水线 + 动态层状态机**。这三个东西一旦从 UI 暴露，竞争对手可以：

| 暴露点 | 对手能反推出 |
|--------|------------|
| 生成步骤标签 | 完整流水线架构（哪些步骤传统算法、哪些 LLM） |
| 动态层面板 | 故事状态机的全部数据结构（锚点/基线/钩子/评分） |
| 可行性百分比 | 置信度评分机制的存 |
| 冲突警告措辞 | 卡间兼容性检测系统 |
| "传统算法+LLM"文案 | 混合架构策略 |

## 二、必须隐藏（Level 1）

### 2.1 生成进度步骤标签

**暴露位置**：
- `moling-workspace-web.html` — 模态框内的 gen-progress 面板
- `moling-mobile.html` — AI生文进度面板

**当前展示**（以 workspace 为例）：
```
权重分配 → 四库过滤 → 冲突检测 → 兼容性打分 → 编织模式匹配
→ 大纲填充 → 后检查 → 动态层更新 → 叙事元素提取
→ 头脑风暴 → 章节写作 → 连贯性校验
```

**泄露等级**：🔴 致命
**泄露内容**：12 步的完整顺序就是算法流程图。竞争对手一眼可知：
- 前 8 步是传统算法（可缓存、确定性的）
- 后 3 步是 LLM 调用
- 存在"动态层更新"作为后处理
- 存在"连贯性校验"作为 QA 门控

**处理方案**：
- 模糊化为 3 个阶段：`预处理中… → 生成中… → 校验中…`
- 每个阶段内部可走进度条动画（给用户确定性反馈），但不展示具体步骤名
- 阶段文案要用自然语言，不带算法术语

### 2.2 动态层面板（Dynamic Layer Bar）

**暴露位置**：
- `moling-workspace-web.html` — 编辑区底部，`#dynamicLayerBar`

**当前展示**：
```
📋 动态层 · 故事坐标系
🎯 POV:林霄 | 地点:演武场 | 时间:第15天·清晨
🔗 must_hold: ✓ 必须保持 / ✗ 必须避免
⚡ 未收束钩子: 神秘人身份(高) / 玉佩秘密(中)
🔄 最近变更: 章14: 林霄↔神秘人接触
📊 可行性评分 92% —— 与动态层高度一致，可生文
```

**泄露等级**：🔴 致命
**泄露内容**：动态层是墨灵的核心技术资产——故事状态机（Story State Machine）。暴露了：
- 锚点维度（POV/地点/时间）→ 数据结构设计
- must_hold / must_not → 连贯性约束机制
- 钩子 + 紧急度分类 → 叙事驱动力引擎
- 可行性评分 → 置信度函数

**处理方案**：
- **完全从 UI 移除该面板**
- JS 中保留动态层数据结构和逻辑（算法仍然用）
- 加入隐藏调试模式（双击章节标题 5 次可调出调试面板）

### 2.3 模态框动态层摘要（modal-dl-summary）

**暴露位置**：
- `moling-workspace-web.html` — `#cardModal` 顶部的 `.modal-dl-summary`

**当前展示**：
```
📋 当前: POV=林霄 | 演武场 · 清晨  | ⚡ 钩子: 神秘人身份(高) · 玉佩秘密(中)
```

**泄露等级**：🔴 致命
**泄露内容**：虽然比完整面板简略，但仍暴露了：
- "POV"作为维度 → 暗示存在锚点系统
- "钩子" + 紧急度 → 暗示存在钩子优先级排序
- 这些术语本身就泄露了内部数据结构

**处理方案**：
- 改为一句无信息的占位文字：`📋 当前上下文已就绪`
- 或者完全移除

## 三、建议模糊化（Level 2）

### 3.1 可行性评分百分比

**暴露位置**：
- `moling-workspace-web.html` — 动态层面板内的 `.feasibility-bar` 和 `.feasibility-score`

**当前展示**：
```
📊 当前卡组可行性
[████████████████░░] 92%
—— 与动态层高度一致，可生文
```

**泄露等级**：🟡 中
**泄露内容**：
- 存在评分函数
- 分值范围（0-100%）
- "与动态层一致"提示 → 评分的判断依据

**处理方案**：
- 动态层面板已移除 → 此项随面板一并移除
- 如需保留生文前检查：改为三档状态灯（🟢 可生文 / 🟡 需调整 / 🔴 不建议），不显示百分比
- 文字改为中性描述

### 3.2 冲突警告

**暴露位置**：
- `moling-workspace-web.html` — 卡牌上的 `.card-conflict-warning`

**当前展示**：
```
⚠️ 与惊艳方向冲突
```

**泄露等级**：🟡 中
**泄露内容**：
- 存在卡间冲突检测算法
- 冲突以方向类型（稳妥/有趣/惊艳）为维度

**处理方案**：
- 改为：`⚠️ 组合偏好提示`
- 或改为 icon-only：`⚠️` 悬停显示"部分方向组合可能影响生成效果"

### 3.3 Landing page 混合架构表述

**暴露位置**：
- `moling-landing-web.html`

**当前展示**：
```
混合流水线引擎（传统算法+LLM），15秒完成整章生成。
```

**泄露等级**：🟡 中
**泄露内容**：
- 公开声明使用"传统算法+LLM"的混合架构
- 让对手知道你的策略方向

**处理方案**：
- 改为：`智能创作引擎，15秒完成整章生成。`
- 去掉"传统算法+LLM"、"混合流水线"等工程术语

### 3.4 卡牌来源标签："Phase 4" 暴露

**暴露位置**：
- `moling-settings-web.html` — 卡牌卡片底部的 `.card-source`
- `moling-settings-mobile.html` — 同上

**当前展示**：
```
📌 Ch.16 · Phase 4
```

**泄露等级**：🔴 致命
**泄露内容**：
- "Phase 4" 是内部工程术语（流水线的第四阶段）
- 竞争对手看到可知系统至少存在 4 个处理阶段
- 每个卡牌标注"Phase 4"暗示了"生成→收纳"的两阶段架构

**处理方案**：
- 改为：`📌 第 16 章 · 收纳生成`
- 只展示用户可理解的章节来源，不展示内部阶段编号
- JS 数据层和 UI 渲染层统一使用友好文案

### 3.5 "新鲜期"机制暴露

**暴露位置**：
- `moling-settings-web.html` — 新鲜期标签 `.fresh-label`, 卡牌徽章 `.fresh-badge`
- `moling-settings-mobile.html` — 同上

**当前展示**：
```
🔥 新鲜期加成（3 章内）     ← 暴露了精确的时间窗口
🔥 新鲜期                    ← 暴露了机制名称
```

**泄露等级**：🟡 中
**泄露内容**：
- "新鲜期加成"暗示存在时间敏感的概率调整机制
- "3 章内"暴露了精确的执行参数
- 竞争对手可推断：新生成的卡牌在某个窗口内有概率优势

**处理方案**：
- 标签改为：`🔥 新推荐（近期生成·优先出现）`
- 徽章改为：`🔥 新牌`
- 文案要传递"这张卡是新生成的"信息，但不说明背后的概率调整逻辑

### 3.6 Phase 4 收纳进度提示

**暴露位置**：
- `moling-workspace-web.html` — 确认收纳后的 toast/反馈

**场景**：用户点击"确认收纳"后，前端触发 Phase 4 异步执行。如果展示"正在更新四库…"，则暴露了"四库是分阶段更新的"架构细节。

**泄露等级**：🟡 中
**泄露内容**：
- "四库"作为内部数据结构名称（虽然已产品化，但加上"更新中"暗示了写操作）

**处理方案**：
- 使用自然语言，不出现架构术语：
  - ✅ `草稿已确认，正在同步世界设定…`
  - ❌ `四库收纳中…`
  - ❌ `Phase 4 执行中…`
- 进度提示只展示用户能理解的因果："你在确认内容 → 系统在更新相关设定"

### 3.7 冷却计数展示

**暴露位置**：
- 抽卡模态框 — 新增的重抽次数指示器

**场景**：当前每次重抽 +1，达到上限后触发 AI 推荐或手动输入。

**泄露等级**：🟢 低
**泄露内容**：
- "冷却"是游戏术语，不直接暴露算法
- 但文案应避免工程术语如"冷却计数""上限阈值"

**处理方案**：
- 使用友好文案：`🔄 今日可重抽 {2}/{3} 次`
- 或：`🔄 还有 1 次重抽机会`
- 避免：`冷却 2/3`、`重抽上限`、`冷却计数`

### 3.8 卡牌池退役/归档

**暴露位置**：
- `moling-settings-web.html` — 新增的 `[可用卡] [已退役] [全部]` 筛选
- `moling-settings-mobile.html` — 同上

**泄露等级**：🟢 低
**泄露内容**：
- "退役"暗示卡牌有生命周期管理，但这是常见的游戏化设计
- 不暴露淘汰算法的具体参数（阈值/上限）

**处理方案**：
- 保留当前文案，属于 Level 4（不应隐藏）
- 但注意不要展示淘汰规则细节，如"common 卡 20 章退役":

### 3.9 四库待补 / 全量重新分析提示

**暴露位置**：
- 四库管理页 — 失败回补角标
- 弹窗 — 连续失败后的建议重新分析

**场景**：Phase 4 连续失败时，用户会看到"有 2 章内容待同步"或"检测到 X 章的四库变更尚未收录"。

**泄露等级**：🟡 中
**泄露内容**：
- "待同步"揭示了四库是按章增量更新的
- "重新分析"暗示了存在批量处理能力
- 但这是合理的用户反馈（用户需要知道数据是否完整）

**处理方案**：
- 使用用户能理解的语言，聚焦于"影响"而非"机制"：
  - ✅ `有 2 章的内容尚未完整收录，建议刷新以确保设定一致`
  - ❌ `2 章 Phase 4 失败，建议手动触发全量重分析`
- 全量重分析按钮文案：
  - ✅ `刷新故事设定`
  - ❌ `全库重新分析`
- 不要出现"Phase 4"、"增量"、"重分析"等内部术语

## 四、低泄露风险（Level 3 — 保留或微调）

| 元素 | 风险 | 理由 | 处理 |
|------|------|------|------|
| 权重滑块 0-100 | 🟢 低 | 常见 UI 模式 | 保留 |
| 单卡/双卡/全选 | 🟢 低 | UX 模式，非算法 | 保留 |
| 卡牌稀有度 ⚪🔵🟣🟠 | 🟢 低 | 游戏化设计 | 保留 |
| 🔄 重抽按钮 | 🟢 低 | 明显 UX 功能 | 保留 |
| 进度条动画（无标签） | 🟢 低 | 用户需反馈 | 保留，去步骤名 |

---

## 五、不应隐藏（Level 4）

| 元素 | 理由 |
|------|------|
| 卡牌内容（标题/描述/标签） | 用户创作内容，非算法 |
| 卡牌选择交互 | 核心操作，必须可见 |
| 权重调节滑块 | 用户控制创作方向 |
| 生成进度条（动画） | 用户需要知道系统在运转 |
| ✅ 确认 / ✗ 拒绝 | 核心编辑交互 |
| "四库"产品命名 | 已经是产品功能名称（四库管理），用户需理解这个概念 |
| 卡牌池退役/归档筛选 | 游戏化设计，不暴露算法参数 |

## 六、"四库"命名的风险边界与安全使用规范

"四库"是一个特殊案例——它既是**产品功能名称**（必须让用户知道），也是**内部架构名称**（暗示了数据结构分为 4 部分）。

### 6.1 允许的使用场景

| 场景 | 示例 | 理由 |
|------|------|------|
| 四库管理页标题 | "四库管理" | 产品功能名称，用户需要知道这个功能 |
| 四库管理页 Tab | "人物库""时间线库""剧情承诺库""世界观库" | 用户需要看到的 4 个分类 |
| 搜索框占位符 | "搜索人物、时间线、剧情承诺、世界观…" | 明确功能范围 |
| 用户引导/新手教程 | "四库会自动记录你的故事设定" | 用户需要理解功能价值 |

### 6.2 禁止的使用场景

| 场景 | 示例 | 原因 |
|------|------|------|
| 进度/状态提示 | "四库收纳中…" | 暗示"四库是分阶段更新的"架构细节 |
| 错误提示 | "四库更新失败，请重试" | 暴露了后台写入操作的存在 |
| 卡牌来源标签 | "Ch.16 · Phase 4" | 直接暴露内部阶段编号 |
| 连续失败弹窗 | "四库已 X 章未更新" | 暴露了每章更新的架构 |

### 6.3 安全替代表

| 想表达的意思 | ✅ 安全文案 | ❌ 危险文案 |
|------------|-----------|-----------|
| Phase 4 正在执行 | "正在同步世界设定…" | "四库收纳中…" |
| Phase 4 执行完成了 | "设定已同步" | "四库已更新" |
| Phase 4 失败了 | "部分章节内容待同步" | "四库更新失败" |
| 连续 X 章失败 | "检测到部分章节内容尚未完整收录" | "四库已 X 章未更新" |

## 七、隐藏调试模式

为保留开发/调试能力，在 UI 中嵌入一个**秘密入口**：

**触发方式**：双击（或 5 次快速点击）编辑器标题区域

**触发后显示**：
- 被隐藏的动态层调试面板
- 实时算法步骤日志
- 注入 Prompt 预览（仅调试用）

**发布前需确认**：
- 调试模式默认关闭
- 没有视觉提示/入口泄露
- 生产构建时可通过 flag 完全移除

## 十、后续维护注意事项

1. **每次 UI 迭代**后检查是否有新的算法信息泄露
2. **社区/用户反馈**中注意是否有人试图反推算法 -> 考虑进一步模糊
3. **竞品发布新功能**时对照检查他们暴露了什么、藏了什么
4. **专利布局**后（如有），部分信息可适度公开以建立技术壁垒
5. **第三方集成**（如有 API）确保不通过接口泄露内部架构
