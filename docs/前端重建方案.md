## 1. 现状审计

### 1.1 当前技术栈

| 维度 | 现状 | 评价 |
|------|------|:--:|
| 框架 | Next.js 15.1 App Router | ✅ 可保留 |
| UI | Radix UI Primitives + Tailwind CSS 4.3 | ✅ 可保留 |
| 状态管理 | 4 个 React Context (Auth/Project/Workspace/SystemHealth) | 🔴 需重构 |
| API 客户端 | 双层架构：新版 apiClient + 旧版 client.ts | 🔴 混乱 |
| 组件库 | ~65 个组件，无统一组件库 | 🔴 缺乏体系 |
| 认证 | localStorage JWT + AuthGuard | 🟡 可用但脆弱 |
| 测试 | Vitest + Playwright，覆盖率低 (~30%) | 🔴 不足 |
| 设计系统 | 8 主题 CSS 变量体系 (`--th-*`) | ✅ 优秀 |

### 1.2 核心问题

| # | 问题 | 影响 |
|---|------|------|
| 1 | 双层 API 客户端混用，无类型安全 | 运行时错误、维护噩梦 |
| 2 | Context 嵌套过深，WorkspaceContext 膨胀 | 不必要的重渲染、难调试 |
| 3 | 组件无清晰分层（ui/business/page 混在一起） | 复用困难、修改波及面大 |
| 4 | 缺少 Server Components 利用 | 全客户端渲染，首屏性能差 |
| 5 | 错误处理不统一（try-catch 散落） | 用户体验不一致 |
| 6 | 无 Loading/Skeleton/Empty/Error 统一策略 | 页面状态管理混乱 |
| 7 | Tailwind 4 与 CSS Modules 混用 | 样式方案不纯 |
| 8 | 移动端为事后适配，非 Mobile First | 触屏体验不佳 |

## 2. 重建策略

### 2.1 核心原则

1. **类型安全第一** — 从 openapi.json 自动生成全部类型，编译期杜绝 API 契约错误
2. **Server First** — 善用 Next.js Server Components + Server Actions，减少客户端 JS
3. **组件原子化** — 按 shadcn/ui 模式构建组件库，每个组件独立、可组合
4. **状态隔离** — 用 Zustand 替代深层 Context，按域拆分 store
5. **Mobile First** — 移动端为第一优先级，桌面端增强
6. **零 CSS Modules** — 纯 Tailwind CSS 4，彻底告别 CSS 文件碎片
7. **全量覆盖** — Loading / Skeleton / Empty / Error 每种状态都实现

### 2.2 保留项

| 保留 | 原因 |
|------|------|
| DESI…已升级为 8 主题 CSS 变量体系 | 专业级设计规范，`--th-*` 变量驱动 |
| Tailwind CSS 4 | 纯 utility classes + `--th-*` CSS 变量 | `var(--th-text)` / `var(--th-bg)` 引用主题色 |
| Radix UI Primitives | 无头 UI 最佳选择 |
| lucide-react 图标库 | 统一图标体系 |
| openapi.json | API 契约的单一真相源 |
| Next.js App Router | 主流方案，无替代必要 |

### 2.3 废弃项

| 废弃 | 替代 |
|------|------|
| CSS Modules (*.module.css) | 纯 Tailwind utility classes |
| 旧版 API 客户端 (src/api/) | openapi-fetch + 自动生成类型 |
| 4 层 React Context | Zustand stores |
| 手写 api.ts (~1500 行) | openapi-ts 自动生成 |
| 手动 Token 管理 | httpOnly cookie 或统一中间件 |
| 页面级 loading.tsx | 组件级 Suspense + Skeleton |

## 3. 技术选型

### 3.1 最终技术栈

| 类别 | 选择 | 版本 | 理由 |
|------|------|:--:|------|
| 框架 | Next.js (App Router) | 16+ | Suspense、Server Components、Partial Prerendering |
| 语言 | TypeScript | 5.7+ | 严格模式 |
| 样式 | Tailwind CSS | 4.x | 已建立设计令牌体系 |
| 组件库 | shadcn/ui 模式 | latest | 复制源码、完全可控、无黑盒依赖 |
| 状态管理 | Zustand | 5.x | 轻量、TS 友好、支持 middleware |
| 数据请求 | TanStack Query | 5.x | 缓存、去重、乐观更新、SSR |
| API 客户端 | openapi-fetch + openapi-ts | latest | 类型安全、自动生成 |
| 表单 | React Hook Form + Zod | 7.x / 3.x | 性能好、验证强 |
| 动画 | framer-motion | 11.x | 声明式、布局动画 |
| 测试 | Vitest + Playwright + Testing Library | latest | 全覆盖 |
| 监控 | Sentry | latest | 已配置 |
| 图标 | lucide-react | latest | 统一图标库 |
| 富文本 | Tiptap (替代当前动态导入方案) | latest | 可扩展、TS 原生 |

### 3.2 选型对比（关键决策）

**状态管理：Zustand vs Redux vs Jotai vs Context**

| 维度 | Zustand | Redux Toolkit | Jotai | React Context |
|------|:---:|:---:|:---:|:---:|
| 打包体积 | 1KB | 11KB | 2KB | 0 |
| TS 支持 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 渲染优化 | 选择器级 | 选择器级 | 原子级 | 无 |
| 中间件 | 丰富 | 丰富 | 有限 | 无 |
| SSR 友好 | ✅ | ✅ | ✅ | ⚠️ |
| 学习曲线 | 低 | 高 | 低 | 极低 |

**结论**：Zustand — 最佳体积/能力/复杂度平衡。

**数据请求：TanStack Query vs SWR vs 手写**

| 维度 | TanStack Query | SWR | 手写 |
|------|:---:|:---:|:---:|
| 缓存策略 | 完整 | 基础 | 需自建 |
| 乐观更新 | ✅ | ✅ | ❌ |
| 无限滚动 | ✅ | ✅ | ❌ |
| 离线支持 | ✅ | ❌ | ❌ |
| DevTools | ✅ | ❌ | ❌ |

**结论**：TanStack Query — 功能最全，生产验证。

### 4.2 数据流架构

```
┌─────────────────────────────────────────────────────┐
│                    Server Side                       │
│  ┌──────────┐    ┌──────────────┐                   │
│  │ RSC Page │───>│ Server Action│ (表单提交)         │
│  └────┬─────┘    └──────┬───────┘                   │
│       │ pre-render       │ fetch                     │
│       ▼                  ▼                           │
│  ┌─────────────────────────────┐                    │
│  │     openapi-fetch client    │                    │
│  │  (types from openapi-ts)    │                    │
│  └─────────────┬───────────────┘                    │
└────────────────┼────────────────────────────────────┘
                 │ HTTP /api/v1/*
                 ▼
┌─────────────────────────────────────────────────────┐
│                    Client Side                       │
│  ┌──────────────────────────────────────┐           │
│  │         TanStack Query               │           │
│  │  ┌─────────┐ ┌──────────┐           │           │
│  │  │ Cache   │ │ Mutation │           │           │
│  │  │ (SSR    │ │ (乐观    │           │           │
│  │  │  hydrate)│ │  更新)   │           │           │
│  │  └─────────┘ └──────────┘           │           │
│  └──────────────┬───────────────────────┘           │
│                 │ 读写                               │
│  ┌──────────────▼───────────────────────┐           │
│  │          Zustand Stores              │           │
│  │  (auth / project / workspace / ui)   │           │
│  └──────────────┬───────────────────────┘           │
│                 │ subscribe                          │
│  ┌──────────────▼───────────────────────┐           │
│  │         React Components             │           │
│  └──────────────────────────────────────┘           │
└─────────────────────────────────────────────────────┘
```

**关键决策**：
- TanStack Query 负责服务端状态（缓存、去重、重试、乐观更新）
- Zustand 负责客户端状态（UI 状态、选中项、表单草稿）
- 两个库边界清晰：服务端数据走 Query，UI 状态走 Store
- openapi-ts 从 openapi.json 自动生成类型，编译期杜绝契约错误

### 4.3 组件设计规范

每个组件遵循统一结构：

```
component-name/
├── index.ts                   # 统一导出
├── component-name.tsx         # 主组件
├── component-name.test.tsx    # 单元测试
└── component-name.stories.tsx # 可选：Storybook
```

每个组件必须覆盖 4 种状态：

| 状态 | 组件 | 行为 |
|------|------|------|
| **Loading** | `<Skeleton />` | 骨架屏，保持布局稳定 |
| **Empty** | `<EmptyState />` | 图标 + 标题 + 描述 + CTA 按钮 |
| **Error** | `<ErrorState />` | 错误信息 + 重试按钮 + 错误详情折叠 |
| **Success** | 正常内容 | 数据渲染 |
