## 文档说明

### 演练目的

1. **验证 RUNBOOK 的有效性**：确保 SOP 步骤正确、完整
2. **提升团队应急响应能力**：让团队成员熟悉故障处理流程
3. **发现潜在问题**：在真实故障发生前发现系统弱点
4. **持续改进**：根据演练结果更新 RUNBOOK

### 演练原则

- **安全第一**：演练不会影响真实用户（使用测试环境）
- **模拟真实**：尽可能模拟真实故障场景
- **记录详细**：记录所有操作和观察结果
- **及时总结**：演练结束后立即总结和改进

### 演练类型

| 类型 | 说明 | 频率 |
|------|------|------|
| **桌面演练** | 团队讨论，模拟故障场景 | 每月一次 |
| **功能演练** | 在测试环境实际执行故障和恢复 | 每季度一次 |
| **全面演练** | 在生产环境执行（谨慎！） | 每年一次 |

### 演练步骤

#### 步骤 1：模拟故障（由演练负责人执行）

```bash
# 方法 1：停止数据库容器（模拟数据库崩溃）
docker stop moling-db

# 方法 2：在数据库内执行慢查询（模拟数据库 CPU 100%）
docker exec -it moling-db psql -U moling -d moling -c "
-- 模拟慢查询
SELECT pg_sleep(60);  -- 休眠 60 秒
"

# 选择一种方法，记录开始时间
echo "故障注入时间: $(date '+%Y-%m-%d %H:%M:%S')"
```

**记录**：
- 故障注入时间：`2026-06-16 10:05:00`
- 故障类型：`数据库停止`
- 注入方式：`docker stop moling-db`

#### 步骤 2：观察系统反应（5 分钟）

```bash
# 1. 查看后端日志
docker logs moling-app --tail 50

# 预期输出：
# sqlalchemy.exc.OperationalError: (psycopg2.OperationalError) could not connect to server

# 2. 查看 API 响应
curl http://localhost:8000/api/health

# 预期输出：
# curl: (7) Failed to connect to localhost port 8000: Connection refused
# 或者返回 500 错误

# 3. 查看前端页面
curl http://localhost:3000/moling/

# 预期输出：
# 页面加载，但数据无法显示（API 调用失败）
```

**记录**：
- 后端错误消息：`sqlalchemy.exc.OperationalError: (psycopg2.OperationalError) could not connect to server`

#### 步骤 4：总结和改进

##### SOP 评估

| SOP 步骤 | 是否清晰 | 是否完整 | 是否有遗漏 | 改进建议 |
|----------|----------|----------|------------|----------|

| 重启数据库 | ✅ 是 | ✅ 是 | ❌ 否 | - |
| 重启后端服务 | ✅ 是 | ✅ 是 | ❌ 否 | - |

##### 发现的问题

1. **问题 1**：SOP 中没有提到需要重启后端服务
   - **影响**：如果只重启数据库，后端仍然无法连接数据库（连接池已断开）
   - **改进措施**：在 RUNBOOK.md 中添加"重启后端服务"步骤
   - **状态**：✅ 已改进

2. **问题 2**：SOP 中没有提到如何验证数据库已完全恢复
   - **影响**：可能数据库容器已启动，但仍在恢复数据中
   - **改进措施**：在 RUNBOOK.md 中添加 `pg_isready` 验证步骤
   - **状态**：✅ 已改进

##### 改进措施

1. **更新 RUNBOOK.md**：
   - 在"故障 2：数据库 CPU 100%" → "恢复步骤"中，添加"重启后端服务"步骤
   - 添加 `pg_isready` 验证步骤

2. **添加自动化脚本**：
   - 创建 `scripts/check_database.sh`：自动检查数据库状态
   - 创建 `scripts/restart_services.sh`：自动重启所有服务

3. **下次演练重点**：
   - 演练"故障 1：API 500 错误"
   - 验证改进后的 SOP 是否有效

## 演练总结

### 成功经验

1. **RUNBOOK 基本有效**：大部分步骤清晰、完整
2. **团队响应迅速**：从故障注入到发现故障只用了 2 分钟
3. **恢复流程顺畅**：重启数据库和后端服务后，系统快速恢复

### 需要改进

1. **SOP 需要更详细**：部分步骤遗漏（如重启后端服务）
2. **需要自动化脚本**：手动执行命令容易出错
3. **需要更多演练**：团队对部分流程不够熟悉

### 改进计划

1. **更新 RUNBOOK.md**（优先级：高）
   - 添加遗漏的步骤
   - 增加更多示例命令
   - 添加预期输出说明

2. **创建自动化脚本**（优先级：中）
   - `scripts/check_health.sh`：检查系统健康状态
   - `scripts/restart_services.sh`：重启所有服务
   - `scripts/backup_database.sh`：自动备份数据库

3. **定期演练**（优先级：高）
   - 每季度一次功能演练
   - 每月一次桌面演练
   - 每年一次全面演练（生产环境）
