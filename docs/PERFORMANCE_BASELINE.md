## 概述

本文档记录了墨灵（Moling）项目的性能基线和压测结果。建立性能基线的目的是：

- 为后续版本的性能对比提供基准
- 识别性能瓶颈和优化机会
- 确保上线后性能稳定
- 支持容量规划和资源分配决策

### 测试范围

- **后端 API**: 所有主要 API 端点
- **数据库**: PostgreSQL 查询性能
- **AI 生成**: LLM 调用性能（如适用）
- **并发用户**: 100 并发用户

## 压测场景

### 场景 A：100 并发用户同时生成文本

**目标**: 测试 AI 生成 API 在高并发下的性能。

**配置**:
- 并发用户数: 100
- 每用户请求数: 10
- 总请求数: 1,000
- 测试时长: 5 分钟

**测试步骤**:
1. 用户登录
2. 获取项目列表
3. 选择项目
4. 调用生成 API（`/api/v1/generate`）
5. 等待响应
6. 重复步骤 4-5

**成功标准**:
- P95 响应时间 < 2,000ms（如调用 LLM）
- 错误率 < 1%
- 吞吐量 > 10 req/s

---

### 场景 B：数据库大量数据查询

**目标**: 测试数据库在大量数据下的查询性能。

**配置**:
- 数据量: 10,000 项目 + 100,000 章节
- 查询类型: 列表查询、详情查询
- 并发用户数: 50
- 测试时长: 3 分钟

**测试步骤**:
1. 查询项目列表（`/api/v1/projects`）
2. 查询章节列表（`/api/v1/projects/{id}/chapters`）
3. 查询卡牌列表（`/api/v1/projects/{id}/cards`）
4. 查询项目详情（`/api/v1/projects/{id}`）
5. 重复步骤 1-4

**成功标准**:
- P95 响应时间 < 500ms
- 错误率 < 0.1%
- 吞吐量 > 100 req/s

## 性能瓶颈分析

### 已识别的瓶颈

#### 1. AI 生成 API 响应时间过长

**现象**:
- P95 响应时间: 1,800ms
- P99 响应时间: 2,500ms

**原因**:
- LLM 调用耗时较长
- 未实现异步生成
- 未使用缓存

**影响**:
- 用户体验较差
- 并发能力受限

**优先级**: 🔴 高

---

#### 2. 章节列表查询性能可优化

**现象**:
- P95 响应时间: 280ms
- 在有 100,000 章节时查询较慢

**原因**:
- 缺少数据库索引
- 未使用分页缓存
- 查询未优化

**影响**:
- 大量数据时性能下降
- 用户体验受影响

**优先级**: 🟡 中

---

#### 3. 项目统计 API 可优化

**现象**:
- P95 响应时间: 140ms
- 需要聚合多个表的数据

**原因**:
- 统计查询未优化
- 未使用物化视图
- 未使用缓存

**影响**:
- 统计页面加载较慢
- 频繁查询时性能下降

**优先级**: 🟢 低

## 改进建议

### 短期改进（1-2 周）

#### 1. 为数据库添加索引

**操作**:
- 为 `chapters.project_id` 添加索引
- 为 `chapters.order` 添加索引
- 为 `card_pool.project_id` 添加索引
- 为 `projects.user_id` 添加索引

**预期效果**:
- 章节列表查询 P95 降低 50%
- 项目列表查询 P95 降低 30%

**责任人**: 后端团队

---

#### 2. 实现 API 响应缓存

**操作**:
- 为项目详情 API 添加 Redis 缓存（TTL 60s）
- 为项目统计 API 添加缓存（TTL 300s）
- 为通知列表 API 添加缓存（TTL 30s）

**预期效果**:
- 缓存命中时响应时间 < 10ms
- 整体吞吐量提升 50%

**责任人**: 后端团队

---

### 中期改进（1-2 月）

#### 3. 优化 AI 生成流程

**操作**:
- 实现异步生成（返回任务 ID，轮询结果）
- 使用 WebSocket 推送生成进度
- 实现生成结果缓存

**预期效果**:
- 用户无需等待生成完成
- 相同提示词命中缓存时 < 100ms
- 并发生成能力提升 5 倍

**责任人**: 后端团队 + AI 团队

---

#### 4. 使用物化视图优化统计查询

**操作**:
- 创建项目统计物化视图
- 定期刷新（每 5 分钟）
- 查询时直接读取物化视图

**预期效果**:
- 项目统计 API P95 < 50ms
- 数据库负载降低 30%

**责任人**: 数据库团队

---

### 长期改进（3-6 月）

#### 5. 实现读写分离

**操作**:
- 配置 PostgreSQL 主从复制
- 写操作走主库
- 读操作走从库

**预期效果**:
- 读性能提升 2 倍
- 写性能不受影响

**责任人**: 运维团队

---

#### 6. 实现分库分表

**操作**:
- 按用户 ID 分库
- 按项目 ID 分表
- 使用数据库中间件

**预期效果**:
- 支持千万级数据
- 查询性能提升 10 倍

**责任人**: 架构团队

### 性能回退检测标准

**自动检测**（通过 CI/CD）:
- P95 响应时间增加 > 20% → ⚠️ 警告
- 吞吐量降低 > 10% → ⚠️ 警告
