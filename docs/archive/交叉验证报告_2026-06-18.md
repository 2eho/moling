## 二、严重问题（4 项，必须修复）

### 🔴 S1: 根目录 `docker-compose.yml` 缺少 Redis 服务

所有文档都假设 Redis 存在（JWT 黑名单、缓存、Celery 队列），但根目录 `docker-compose.yml` 只有 `db/app/web` 三个服务，没有 `redis`。

| 文档 | 声明 | 实际 |
|------|------|------|
| ARCHITECTURE.md | Redis 7, 容器 `moling-redis`, 端口 6379 | ✅ |
| DEPLOYMENT.md | Redis 7, 端口 6379, 缓存+消息队列 | ✅ |
| 根目录 `docker-compose.yml` | 无 redis 服务 | ❌ 缺失 |

### 🔴 S2: 两套 Docker Compose 定义不一致

根目录 `docker-compose.yml` 与 `docker/docker-compose.yml` 服务定义完全不同：

| 维度 | 根目录 | `docker/` 子目录 |
|------|--------|-----------------|
| 数据库镜像 | `postgres:16-alpine` | `pgvector/pgvector:pg17` |
| 前端服务名 | `web` / `moling-web` / 3000 | `frontend` / `moling-frontend` / 80+443 |
| 后端容器名 | `moling-app` | `moling-api` |
| 端口暴露 | `ports`（暴露宿主机） | `expose`（仅内部） |
| 网络名 | `moling-net` | `moling-network` |
| Redis | 无 | 有（带密码持久化） |

### 🔴 S3: ARCHITECTURE.md 完全缺失 Phase 4 状态机

SPECIFICATIONS.md 定义了完整的 Phase 4 状态机（`IDLE→QUEUED→LOCKING→EXTRACTING→VERIFYING→MERGING→COMMITTING→DONE`）、分布式锁（Redis SET NX EX, TTL 30s）、三层幂等性，但 ARCHITECTURE.md 中完全没有提及。

### 🔴 S4: 网络名不统一

| 来源 | 网络名 |
|------|--------|
| 根目录 `docker-compose.yml` | `moling-net` |
| `docker/docker-compose.yml` | `moling-network` |
| ARCHITECTURE.md | `moling-network` |
| RUNBOOK.md | `moling-network` |

RUNBOOK.md 的回滚命令使用 `--network moling-network`，按根目录 `docker-compose.yml` 部署时会因网络不存在而失败。

## 三、中等问题（4 项，建议修复）

### 🟡 M1: GIT_WORKFLOW_GUIDE.md CI 章节描述与实际脱节

文档说"审查了 6 个 workflow 文件"、"建议合并为 3 个"、"大量 `|| true`"，但实际：
- 当前只有 1 个 `ci.yml`（已合并完成）
- 实际 `ci.yml` 没有使用 `|| true`
- 包含 Bandit 安全扫描和 flake8（文档未提及）
- 前端 Lint 使用 `continue-on-error: true`（非阻断）

### 🟡 M2: RUNBOOK.md 混用 `docker-compose` 和 `docker compose` 命令

DEPLOYMENT.md 统一使用新版 `docker compose`（v2），但 RUNBOOK.md 同时使用旧版 `docker-compose`（v1）和新版。

### 🟡 M3: RUNBOOK.md 健康检查路径缺少 `/v1`

RUNBOOK.md 用 `/api/health`，DEPLOYMENT.md 用 `/api/v1/health`。取决于后端路由，可能一个正确一个 404。

### 🟡 M4: ONBOARDING.md 第 177 行命令不可用

```bash
docker-compose up -d db redis
```
根目录 `docker-compose.yml` 没有 `redis` 服务，这条命令会失败。
