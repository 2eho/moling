# 墨灵前端推翻重做 — 需求规格

> 版本: v1.0 | 日期: 2026-06-19 | 状态: 待确认

---

## 1. 问题陈述

### 1.1 为什么要重做

当前前端 (`moling-web`) 存在以下结构性缺陷，修修补补已无法从根本上解决：

| # | 问题类别 | 严重度 | 根因 |
|---|---------|--------|------|
| 1 | **样式分裂** | 🔴 致命 | Tailwind 4 + 52个CSS Module文件并存，同一组件同时用两套方案 |
| 2 | **巨型文件** | 🔴 致命 | `api.ts` 940行、`mock/index.ts` 964行、`globals.css` 771行，单体巨石 |
| 3 | **认证守卫三重奏** | 🟡 高危 | AuthGuard组件 / 内联useEffect / 条件渲染 — 三种模式行为不一致 |
| 4 | **Mock路径与真实API脱节** | 🟡 高危 | Mock注册路径与实际API路径不匹配，离线开发系统形同虚设 |
| 5 | **Zod闲置** | 🟡 高危 | `apiValidation.ts` 定义完整但从未集成到API调用中 |
| 6 | **Context肥胖** | 🟡 高危 | `WorkspaceContext` 338行混合UI状态、轮询、数据、兼容存根 |
| 7 | **死代码积累** | 🟡 高危 | `useApi`、`usePolling`、`loadChapter`存根等 |
| 8 | **OpenAPI类型未生成** | 🟡 高危 | 有脚本 `openapi:generate` 但 `api-types.ts` 不存在 |
| 9 | **全客户端组件** | 🟡 建议 | 18个路由全部Client Component，首屏无SSR/SSG |
| 10 | **API路径硬编码** | 🟡 建议 | 无共享路由常量，修改成本高 |

### 1.2 范围

- ✅ 重写：`moling-web/src/` 下所有前端代码
- ✅ 保留：后端API（不变）、数据库模型（不变）、部署配置（不变）、`.github/`（不变）
- ❌ 不涉及：`moling-server/`、`docker/`、`docs/`、`novel_dissector/`

## 2. 用户故事

### US-1: 统一的视觉体验
> 作为用户，我希望整个产品的视觉风格一致，所有页面看起来像同一个产品，而不是东拼西凑的。

**验收标准**：
- 仅使用 Tailwind CSS 4，0个 CSS Module 文件
- 设计令牌（颜色/字体/阴影/间距/动画）统一在 `globals.css` 的 `@theme` 中定义
- 所有组件使用相同的设计语言（按钮、输入框、卡片、模态框风格一致）
- 深色/浅色主题切换流畅，所有组件支持双主题

### US-2: 流畅的工作台体验
> 作为作者，我希望在工作台（workspace）中快速完成创作、抽卡、AI生成、查看四库等操作，不需要频繁等待加载。

**验收标准**：
- 三栏布局（左侧四库 + 中间编辑器 + 右侧AI工具箱）拖拽流畅（≥60fps）
- 章节切换 < 500ms
- AI生成进度实时可见（轮询 ≤ 2秒）
- 抽卡动画流畅（≥60fps）

### US-3: 可靠的认证体验
> 作为用户，我希望登录后访问任何页面都有统一的认证保护，不会出现某些页面裸奔的情况。

**验收标准**：
- 所有需认证路由使用统一的认证守卫模式（Next.js Middleware）
- Token过期自动静默刷新，用户无感知
- 登出清除所有本地状态
- 记住我功能正常工作

### US-4: 真实数据的四库管理
> 作为作者，我希望四库（角色/时间线/伏笔/世界观）的数据来自真实后端，而非前端的假数据。

**验收标准**：
- 所有数据展示（角色库、时间线、伏笔库、世界观、卡牌池）调用真实API
- Loading/Empty/Error 三态覆盖所有数据组件
- 搜索、过滤、分页功能完整

### US-5: 离线开发能力
> 作为前端开发者，我希望在没有后端的情况下也能完整开发和调试前端功能。

**验收标准**：
- `NEXT_PUBLIC_MOCK_ENABLED=true` 时启用完整Mock
- Mock端点的路径与实际API路径完全一致
- Mock数据足够丰富（≥5个项目、≥3个章节、≥8张卡牌、四库数据齐全）
- Mock模式下无需任何后端即可走通全流程

### US-6: 类型安全的API调用
> 作为开发者，我希望API调用的请求和响应都有完整类型约束，编译器能帮我发现接口变更。

**验收标准**：
- 基于 `openapi.json` 自动生成 TypeScript 类型（`api-types.ts`）
- 所有API函数使用生成的类型
- Zod schema在API响应层自动验证
- CI流程中 `openapi:check` 检测类型漂移

## 3. 约束与规则

| 约束 | 说明 |
|------|------|
| **技术栈不变** | Next.js 15 (App Router) + React 19 + TypeScript 5.7 + Tailwind CSS 4 |
| **子路径部署** | `basePath: "/moling"`，所有资源路径带前缀 |
| **后端API不变** | 150+端点接口不改变，前端重写不涉及后端 |
| **生产级品质** | 所有组件覆盖 Loading/Empty/Error/Edge Case 四态 |
| **移动端兼容** | 所有页面支持移动端自适应 (375px – 1440px) |
| **测试覆盖** | 核心组件 ≥80% 测试覆盖，关键流程 E2E 覆盖 |
| **渐进交付** | 分批次交付，每批独立可部署，不阻塞后端 |

---

## 4. 非目标（不做什么）

1. ❌ 不增加新功能（纯粹的等同功能重建）
2. ❌ 不改变后端API
3. ❌ 不改变数据库
4. ❌ 不改变部署配置
5. ❌ 不引入新的第三方UI库（保持 Radix UI 作为唯一Headless库）
6. ❌ 不使用 Tailwind 之外的CSS方案（如 styled-components、CSS-in-JS）
