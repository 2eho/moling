## 一、概述

### 1.1 测试目标

| 目标 | 描述 |
|:----|:-----|
| **功能正确性** | 所有 63 个 API 端点行为符合接口文档定义 |
| **数据一致性** | 前后端数据交互无丢失、无篡改、无格式错误 |
| **错误处理** | 所有异常场景返回正确 HTTP 状态码和业务错误码 |
| **边界条件** | 空值、极值、非法输入均有明确处理逻辑 |
| **性能基线** | API P95 响应时间 < 500ms（非 LLM 调用），生成任务 < 30s |
| **安全合规** | 认证鉴权、SQL 注入、XSS、敏感数据暴露均无漏洞 |

### 1.2 测试范围

**包含范围**：

- ✅ 后端 API 全部 63 个端点（P0: 24 + P1: 10 + 管理后台）
- ✅ 前端 11 个页面所有交互流程
- ✅ 数据库 CRUD 操作及事务一致性
- ✅ Redis 缓存一致性
- ✅ Celery 异步任务执行正确性
- ✅ LLM 调用的 Mock 及集成测试
- ✅ 前后端联调（API 请求/响应对齐）

**不包含范围**（v1.0）：

- ❌ 负载均衡和高可用测试（MVP 单机部署）
- ❌ 多浏览器兼容性测试（MVP 仅支持 Chrome/Edge 最新版）
- ❌ 移动端适配测试（MVP 仅支持桌面端）
- ❌ LLM 输出质量评估（AI 生成内容的主观质量）

### 1.3 测试策略

```
测试金字塔模型（墨灵适配版）

        /\
       /E2E\        ← 端到端测试（10%）：核心业务流程
      /______\
     /        \      ← API 集成测试（25%）：接口契约测试
    /__________\ 
   /            \     ← 服务层单元测试（35%）：业务逻辑
  /______________\  
 /                \   ← 数据层单元测试（30%）：DAO/模型
/________________\
```

**测试分层策略**：

| 层级 | 框架 | 覆盖目标 | 执行频率 |
|:-----|:-----|:---------|:----------|
| **数据层** | pytest + SQLAlchemy | Model 验证、DAO CRUD、事务回滚 | 每次提交 |
| **服务层** | pytest + unittest.mock | 业务逻辑、LLM 调用 Mock、异常处理 | 每次提交 |
| **API 层** | httpx + pytest | 端点请求/响应、认证鉴权、错误码 | 每次提交 |
| **前后端集成** | Playwright | 登录→建项目→进工作台→抽卡→生成→收纳全流程 | Daily |
| **性能** | Locust | API 并发、生成任务队列、数据库查询 | Weekly |
| **安全** | bandit + manual | SQL 注入、XSS、认证绕过 | Per Release |

## 三、测试框架选型

### 3.1 后端测试框架

| 框架 | 用途 | 版本 | 选型理由 |
|:-----|:-----|:----:|:---------|
| **pytest** | 单元测试 / 集成测试 | ≥8.3 | 生态最成熟，Fixture 系统灵活，asyncio 支持好 |
| **pytest-asyncio** | 异步测试支持 | ≥0.24 | FastAPI 异步端点必需 |
| **httpx** | API 测试客户端 | ≥0.28 | 支持 ASGI 传输（直接测试 FastAPI） |
| **pytest-cov** | 覆盖率统计 | ≥5.0 | 生成覆盖率报告，集成 CI |
| **factory-boy** | 测试数据工厂 | ≥3.3 | 创建可复用的测试数据模板 |
| **bandit** | 安全扫描 | ≥1.7 | 静态安全扫描，检测常见漏洞 |

### 3.2 前端测试框架

| 框架 | 用途 | 版本 | 选型理由 |

| **@testing-library/react** | React 组件测试 | ≥16.0 | 官方推荐，测试用户行为而非实现细节 |

### 6.3 事务一致性测试（Phase 4 关键）

```python
# moling-server/tests/test_services/test_phase4_service.py
class TestPhase4Transaction:
    """Phase 4 事务一致性测试。"""
    
    async def test_phase4_rollback_on_error(self, test_db, test_project, test_chapter):
        """Phase 4 失败时事务应回滚（数据库状态不变）。"""
        # Mock LLM 调用失败
        with pytest.mock.patch("app.llm.client.LLMClient.generate") as mock_generate:
            mock_generate.side_effect = Exception("LLM API Error")
            
            with pytest.raises(Exception):
                await phase4_service.execute_phase4(
                    test_db, test_project.id, test_chapter.id
```

### 14.2 常见测试陷阱

| 陷阱 | 描述 | 规避方法 |
|:-----|:-------|:---------|
| **测试顺序依赖** | 测试用例 B 依赖用例 A 的执行结果 | 每个用例独立准备测试数据 |
| **未清理测试数据** | 上次测试的数据影响下次测试 | 使用 Fixture 的 `yield` + 清理逻辑 |
| **Mock 不彻底** | Mock 了 LLM 但忘了 Mock Redis | 使用统一 Mock 装饰器 |
| **异步测试遗漏 await** | pytest 显示通过但实际未执行断言 | 使用 `pytest-asyncio` 并标记所有异步测试 |
